# Vault Operator v2.14: Smarter retrieval, controllable reasoning

v2.14 is a feature wave. It sharpens how the plugin finds notes (the Retrieval Wave 1 work, inspired by a close read of Gbrain's pipeline) and adds a per-conversation control surface for how hard the model thinks: a binary thinking toggle and a model-native reasoning-effort slider that reach every default provider, including Amazon Bedrock. Alongside the two headline areas it carries a batch of provider and routing fixes that came out of community issues and live testing.

## v2.14.15 -- Quieter discovery logs for unconfigured local providers

Tiny cosmetic patch. The "ollama-main unreachable (skipping refresh)" debug line landed on every plugin reload for users who had an Ollama or LM Studio provider configured in the past but did not have the local server running. Already on `console.debug` (not `console.warn`), but DevTools-verbose users saw it twice per reload.

`ModelDiscoveryService` now rate-limits the unreachable-debug to once per provider per session. The set resets when a non-network error happens for the same provider so a real failure still surfaces after the user touches the config.

No behaviour change otherwise. This closes the last cosmetic item the v2.14.0 stability audit flagged.

## v2.14.14 -- Audit refactoring sweep -- the v2.14 line is stable

Closes every refactoring recommendation from the v2.14.0 stability audit plus the two outstanding IMPs (PDF reindex UI, Stufe-2 click pipeline). After v2.14.9..13 shipped all P0 and P1 defects, this release works through the audit's roadmap of structural cleanups so the v2.14 line lands on a foundation you can build on.

- **Confirm-Modal for destructive actions (REF-01).** Three delete buttons (model removal, conversation delete, custom agent delete) now route through `confirmModal` with a destructive primary action. Closes the "destructive actions need confirmation" rule the audit cited.
- **Status truth for FEAT-19-22 (REF-02).** The Aktiver-Dialog-Ingest entry in the backlog was flipped from Done/Released to Partial/Active. The implementation is a 5-paragraph picker, not the multi-turn dialog the feature claimed -- the marker now matches the code.
- **TaskRouter.classifyWithFallback removed (REF-03).** Dead since v2.10.x; no production caller. Removed along with its two unit tests. The regex-only classifier stays authoritative.
- **Always-on meta-tools in cachedTools (REF-04).** `find_tool` and `read_skill` now ride through `cachedTools` even when a custom agent excludes the `agent` group, so progressive disclosure does not silently break on profile-restricted runs.
- **PDF reindex UI (IMP-06-01-01).** PDF embeddings created before v2.14.10 carry the "PDF Parser is not installed" placeholder string. New `SemanticIndexService.reindexPdfsOnly()` plus an EmbeddingsTab CTA (visible until the user runs it) and a one-shot `PdfReindexHintModal` on layout-ready clean those up.
- **MultiFileAtomicCommit removed (REF-05).** Implemented for FIX-15-00-01 but never wired into production. Removed; KnowledgeDB and MemoryDB each ship their own journal-based atomic write.
- **Shared OpenAI-Chat-Completions user-message helper (REF-06).** Three near-identical user branches in openai/copilot/kilo-gateway now delegate to `appendOpenAiChatUserMessage()`. Future image-block / tool-result drift can no longer hide in one of three places.
- **GitCheckpointService.cleanup() is real (REF-07).** Pre-fix the method was a `console.debug` no-op. It now prunes loose `refs/heads/task-*` and `refs/heads/vault-health-*` branches older than 30 days, throttled to once per 24h. Only ref types the plugin itself creates are touched.
- **MemoryTab + FirstRunWizardModal on providerConfigs[] (REF-08).** The atomiser model dropdown in MemoryTab and the "how many LLM models" line in the wizard both read providerConfigs[] now (post-EPIC-26 canonical store), with the legacy activeModels[] kept as a fallback for pre-migration installs. Helper `expandProviderConfigsToCustomModels` bridges the two shapes read-only.
- **SkillSnapshotService binary-aware (REF-09).** The adapter interface now carries optional `readBinary`/`writeBinary`. A path-based extension classifier routes images, archives, PDFs, Office files, fonts, audio and video through the binary methods. Skills with binary assets snapshot and restore byte-faithfully.
- **DynamicToolLoader.loadAll retired (REF-10).** FEAT-29-06 SC-02/SC-03 claimed the legacy custom_* tool runtime was gone, but the loader still ran on every startup. `loadAll()` and its ToolRegistry/ISandboxExecutor imports were removed; `migrateToSkills()` stays for one-shot migration of any remaining legacy data.
- **Tests for IgnoreService, OperationLogger, ConsoleRingBuffer (REF-11).** Three central governance / observability components had zero coverage. Added 32 regression tests across the three files: fail-closed behaviour, always-blocked / always-protected paths, gitignore pattern matching, ReDoS guards, parameter sanitisation (sensitive keys, file-content length, URL userinfo stripping, overlong values), ring buffer overflow, query filters, currentTool labelling, install/uninstall symmetry.
- **Stufe-3 hook construction extracted (REF-12).** The 130-line block of inline lambdas in `main.ts` that wired `preFilter / webUpdatePass / notificationSink / budgetExceededSink` is now a single `buildStufe3Hooks()` call. The hooks live in `src/core/health/Stufe3Hooks.ts` behind a three-field host interface; main.ts has a 7-line dynamic import + call site where 80 lines of inline lambdas used to live.
- **Stufe-2 click drives the pipeline (IMP-19-19-01).** Clicking the freshness hint no longer opens a second tipp-notice with a string to paste manually. It now calls `sendMessageToAgent()` with a focused anti-echo prompt; the sidebar activates and the request goes through the full ToolExecutionPipeline (approval + checkpoint + result delivery), which is what the audit asked for.

Test suite: 2996 passing plus 1 expected fail (+27 vs v2.14.13 from the new IgnoreService / OperationLogger / ConsoleRingBuffer coverage). Type check clean. Build clean.

With this, every P0, every P1, and every refactoring recommendation from the v2.14.0 stability audit is released or explicitly retired. Only the cosmetic double-init of the API handler on plugin load stays open as a P3 follow-up.

## v2.14.13 -- Office tools land in the right auto-approval group

Final cleanup of the v2.14.0 stability audit. The companion to FIX-29-99-01 (which fixed `TOOL_GROUP_MAP` in `builtinModes.ts`, the agent-visible tool schema) closes the same drift on a separate map: `TOOL_GROUPS` in `ToolExecutionPipeline.ts`, which decides which auto-approval bucket a tool falls into. Tools not listed there silently defaulted to `note-edit`, which was correct for most cases but wrong for the four Office binary creators (`create_pptx`, `create_docx`, `create_xlsx`, plus `create_excalidraw`, `create_drawio`, `extract_zip`) and several meta-tools.

Concrete reclassifications:
- Office binary writers and `restore_checkpoint`, `vault_health_check` -> `vault-change` (they create or restore structural artifacts, not just edit a note).
- Read-only meta-tools (`read_document`, `list_checkpoints`, `read_checkpoint`, `diff_checkpoint`, `read_skill`, `read_mcp_tool`, `recall_memory`, `inspect_self`, `search_history`) -> `read`.
- Discovery and routing helpers (`find_tool`, `plan_presentation`, `consult_flagship`, `mark_for_memory`) -> `agent`.
- Skill/MCP entry points (`invoke_skill`, `invoke_mcp_server`, `probe_plugin`, `run_skill_script`) -> `skill`.
- `update_soul` -> `self-modify` so the user is always prompted before the agent identity is rewritten.

Test suite: 2971 passing + 1 expected fail. Type check clean. Build clean.

With this, every P0 and P1 from the v2.14.0 stability audit is either released or explicitly deferred as an IMP (`IMP-19-19-01` for the Stufe-2 click pipeline; `IMP-06-01-01` for one-time PDF reindex; cosmetic double-init of the API handler as P3).

## v2.14.12 -- Stability-audit sweep: six cluster fixes plus drift guard

Sweep release that closes the remaining P1 findings from the v2.14.0 stability audit. After three single-issue hotfixes (v2.14.9 MCP, v2.14.10 PDF, v2.14.11 Stufe-2 SQL), this bundle picks up the structural and cluster-level issues so the v2.14 line lands on a stable foundation before further feature work.

- **TOOL_GROUP_MAP drift structurally closed (FIX-29-99-01).** Eight tools that shipped over time -- `anti_echo_search`, the three memory-source tools (`mark_note_as_memory_source` / `unmark_note_as_memory_source` / `list_memory_source_notes`), `find_tool`, `read_skill`, `probe_plugin`, `run_skill_script` -- were registered in `ToolRegistry` but never added to a group, so the agent could only reach them by hallucination. The eight are now wired into the correct groups, and a new auto-discovery test parses the `ToolName` union out of `types.ts` and refuses any future type-level entry that does not appear in either `MUST_BE_REACHABLE`, `INTENTIONALLY_NOT_REACHABLE`, or `INTERNAL_TOOLS`. The BUG-021 drift pattern -- five repetitions before this guard landed -- is now compiler-enforced.
- **Vault health cluster (FIX-19-99-02).** Five related defects bundled. Stufe-3 lived behind `autoTrigger.enabled` and lost its lastRun marker on every reload; it now has its own `stufe3PeriodicJob` setting plus persisted `lastRunIso` with a six-day margin so the weekly cadence survives restarts. The `strongSignal` heuristic counted raw URLs (so two links to the same outlet flipped the flag); it now groups by eTLD+1 surrogate and demands three distinct domains. The `FreshnessFrontmatterPatcher` was implemented but never instantiated; the orchestrator now wires it through three optional dependencies (`frontmatterPatcher`, `writeFrontmatterEnabled`, `getFileByPath`) so the `freshness.writeFrontmatter` setting finally has an effect, and the allowlist (`freshness:` only) per ADR-95 keeps the write surface narrow. The `vault_health_check` tool's mutating actions (`fix_backlinks`, `cleanup`, `fix_categories`) flipped `isWriteOperation` to true and now take a multi-file snapshot of the whole vault before running, so mass frontmatter edits gain a restore point. And `checkMissingBacklinks` learned cross-property reciprocity: `Quelle.Notizen -> Konzept` counts as satisfied when `Konzept.Quellen -> Quelle` exists (configurable via `vaultHealth.reciprocalProperties`, default `[['Notizen', 'Quellen']]`).
- **Settings deeplinks, ModeService exposure, OnboardingFlow (FIX-26-99-01).** `openSettingsAt('agent', ...)` was passing a tot-string that was not in the `TabId` union, so the EPIC-26 migration modal and the memory-upgrade modal landed on whichever default tab happened to be open. The signature is now `TabId`-typed and the URL deeplink handler and `UpdateSettingsTool` reject unknown tabs explicitly. The `ModesTab` and `NewModeModal` were instantiated with `undefined` as the `ModeService`, so the settings tab could enumerate built-in modes but not write through to the running service; `AgentSidebarView` now exposes the live service via `getModeServiceOrNull()` and the settings tab looks it up on render. The `OnboardingFlow` setup button was opening a `ModelConfigModal` whose save pushed into the post-EPIC-26 dead `activeModels[]` array; the button now routes straight to the providers tab.
- **Provider and sandbox cluster (FIX-04-99-01).** The ChatGPT-OAuth (Codex) path joined the same image-block class that FIX-04-03-09 had closed for the OpenAI / Copilot / Kilo providers -- vision queries returned text only, the model said "I don't see an image". The Responses-API content shape now carries an `input_image` variant and the user-message branch emits it. `SemanticSearchTool` gained a post-fetch `IgnoreService` filter so paths marked in `.obsidian-agentignore` no longer leak through the semantic arm. The iframe sandbox bridge mis-handled two cases on mobile: `vault.list` returned an un-resolved Promise that `postMessage` could not structured-clone (silent empty result), and `vault.mkdir` had no proxy at all even though `SandboxBridge.vaultMkdir` was already implemented. Both are wired now, so the skill-creator skill no longer hits "vault.mkdir is not a function" on iPhone.
- **Office pipeline cleanup (FIX-29-99-04).** Two P0s on the same root cause (the half-implemented ADR-048 PPTX pipeline). The bundled `office-workflow` skill referenced `ingest_template` -- a tool that did not exist; the skill is now a 5-step adhoc-only flow that explicitly tells the agent corporate-template ingestion is not available today, and the lingering `ingest_template` references in `TemplateCatalog` warnings and the `create_pptx` description are gone. Independently, the `plan_presentation` -> `create_pptx` pipeline produced empty slides because the planner emitted `source_slide` + `content: {ShapeName: value}` and the renderer only knew the flat `title`/`body`/`bullets` shape; a new `adaptDeckPlanSlide()` folds the template shape into the adhoc one (Title / Headline / Heading / Slide_Title into title, Subtitle / Subheadline into subtitle, Bullet* sorted into bullets, Body / Content / Description into body with a join-of-remainders fallback) so the plan output can be copied verbatim into `create_pptx`. Eight regression tests pin the heuristic.
- **Tension detection and ModelDiscoveryService (FIX-19-99-03).** `FEAT-19-13` shipped as Released but `IngestDeepTool` had `const tensionDetector: TensionDetector | undefined = undefined` hardcoded -- the detector was never built and the deep-ingest pipeline always skipped marker generation. The detector is now constructed when both the semantic index and a helper-api handler are available (graceful undefined fallback otherwise) and uses the helper budget for the classify call. `FEAT-26-02` similarly shipped Released but `ModelDiscoveryService` received only `{id, displayName}` from `fetchProviderModels`; pricing and capability fields that OpenRouter ships inline (`pricing.prompt`, `pricing.completion`, `top_provider.context_length`, `top_provider.max_completion_tokens`) are now forwarded through the new `FetchedModelEntry` shape into `RawDiscoveredModel`.

Test suite: 2971 passing plus 1 expected fail (+10 vs v2.14.11 from the new auto-discovery, adapter, and Stufe-2 tests). Type check clean. Build clean.

Of the original five P0 findings, four shipped in v2.14.9 / v2.14.10 / v2.14.11 / v2.14.12 (`plan_presentation` schema and `ingest_template` retired this release); the remaining `plan_presentation` outline generator is kept registered as an opt-in helper but is no longer required by the bundled skill. ## v2.14.11 -- Stufe-2 knowledge freshness trigger fixed

Third hotfix from the v2.14.0 stability audit. The Stufe-2 activity trigger -- the dezent inline hint that should fire when you open or modify a note in a cluster whose freshness score has decayed -- was effectively dead since FEAT-19-19 landed. The audit traced it back to two converging defects.

- **FIX-19-19-01: invalid nested aggregate SQL.** `computeAvgAge` issued `SELECT AVG(MAX(mtime)) FROM (SELECT path, MAX(mtime) AS mtime FROM vectors WHERE path IN (?) GROUP BY path)`. The inner subquery already collapses to one max-mtime row per path via the `GROUP BY`, so the outer `AVG(MAX(...))` is a nested aggregate that sql.js correctly rejects with `misuse of aggregate function MAX()`. The fix is `AVG(mtime)` on the subquery. Verified against an in-memory sql.js DB with three data points.
- **FIX-19-19-01: missing error guard in `maybeHint`.** Every DB call inside the hint path was unprotected. A throw inside `computeAvgAge` (or any future DB error) escaped through `void this.maybeHint(file)` in the vault-event hooks as an unhandled rejection, silent to the user but visible in the console. The whole body now lives inside a top-level try/catch with `console.warn` + `return false`, so the hint pipeline contains its own errors and never blocks the editor.
- **Two regression tests pin the contract.** A new `Stufe2ActivityTrigger.test.ts` covers (1) `computeAvgAge` against a real sql.js vectors table returning a finite positive day-value, and (2) `maybeHint` returning `false` instead of throwing when the underlying DB call fails. Both tests RED against pre-fix code, GREEN after the fix.

Test suite: 2960 passing plus 1 expected fail. Type check clean. Build clean.

The audit's live-verification pass against this hotfix surfaced one additional vault-health finding worth tracking separately: the missing-backlinks check reports a single high finding (`Notes/Zettelkasten-Workflow.md`) where the link relationship is semantically reciprocal but routed through two different frontmatter properties (`Quelle.Notizen` -> Konzept, `Konzept.Quellen` -> Quelle). The current predicate expects identical-property reciprocity. Cross-property reciprocity is a separate spec.

## v2.14.10 -- PDF parsing fixed across ingest, mirror, and semantic index

Hotfix for [issue #36](https://github.com/pssah4/vault-operator/issues/36) and the broader regression surfaced by the v2.14.0 stability audit: all PDFs returned the placeholder text `(PDF Parser is not installed...)` even when the `pdfjs-bundle.js` Optional Asset was installed.

The reporter saw the placeholder in attachment previews. The audit found the same placeholder was leaking into the semantic index (vector embeddings), the PDF markdown mirror, both deep-ingest paths, and the agent's `read_document` tool. Root cause: `parseDocument` declared the plugin reference as an optional parameter, so the TypeScript compiler could not flag the five call-sites that quietly passed nothing. `parsePdf` then short-circuited to the placeholder branch and the result fell through into long-term storage.

- **Type contract closed.** `parseDocument(data, ext, plugin)` and `parsePdf(data, plugin)` now require the plugin argument. The compiler refuses any future call that drops it.
- **All five call-sites carry the plugin.** `ReadDocumentTool`, `IngestDocumentTool`, `IngestDeepTool` (via `readSourceAsMarkdown` and a freshly-typed `DeepIngestPipeline` constructor), `PdfMarkdownMirror` (constructor now takes the plugin), and `SemanticIndexService` (new `plugin` option on `SemanticIndexOptions`). `main.ts` wires the plugin into the service at construction; the service's `extractDocumentText` defensively short-circuits with a `console.warn` when the plugin is missing, instead of silently embedding the placeholder.
- **Three regression tests pin the parser behaviour.** New `PdfParser.test.ts` covers the two NOT_INSTALLED branches (missing `bundleLoader`, loader returning `null`) and verifies the loader is called exactly once. Twenty-eight existing tests across `parseDocument`, `SourceReader`, `DeepIngestPipeline`, and `AttachmentHandler` were updated to pass a stub plugin where the runtime never reaches PDF parsing.

One-time index repair is not part of this release: the audit recommendation to add a "Reindex PDFs only" Settings action with a hint modal is tracked separately and will land in a follow-up. Existing PDF embeddings stay corrupted until the user triggers a reindex; new embeddings from this release forward will be correct.

Test suite: 2958 passing plus 1 expected fail. Type check clean. Build clean.

## v2.14.9 -- MCP indirect prompt injection + soul.md leak fixed

Three converging fixes against [issue #46](https://github.com/pssah4/vault-operator/issues/46): a Vault Operator user on Claude Desktop saw the client flag the MCP server's first tool response as indirect prompt injection. Adversarial multi-agent verification confirmed the underlying problem was real and that the Claude Desktop refusal was correct.

- **FIX-23-09-01: stop auto-injecting soul.md and user memory into MCP tool responses.** The bridge silently appended `buildPrompts()` to the first tool response of every session. The prompt body itself was written in imperative wording ("CRITICAL", "NON-NEGOTIABLE", "MUST", "ALWAYS", "FINAL action even if the user says goodbye") and embedded the local agent persona from `soul.md` plus the full user memory dump (profile + patterns + projects). Tool responses are now returned verbatim. The systemContext payload was rewritten in neutral, non-imperative language with the persona and memory sections removed; persona and memory remain reachable only through the explicit `save_to_memory` / `recall_memory` tools the user opted into.
- **FIX-23-09-02: hardened the `execute_vault_op` deny-list.** The issue reporter named seven tools that should not have been externally invokable via the catch-all: `execute_command` (arbitrary Obsidian commands), `use_mcp_tool` + `invoke_mcp_server` (polymorphic MCP dispatch), `invoke_skill` + `run_skill_script` (arbitrary skill code), `evaluate_expression` (JS eval in the sandbox), and `update_soul` (identity mutation). All seven are now in `AGENT_INTERNAL_TOOLS`, so they are filtered from the externally visible tool catalog, the `execute_vault_op` boundary check, and the `get_context` "Available Operations" listing in a single place.
- **FIX-23-09-03: relay path reaches parity with the local HTTP path.** After 23-09-01 renamed the prompts to neutral identifiers, claude.ai connector users could not discover them because `RelayClient` re-implemented JSON-RPC routing and only handled `initialize`, `tools/list`, `tools/call`, and `resources/list` -- returning `{}` for `prompts/list` and `prompts/get`. Adversarial verification also caught two pre-existing transport-shape drifts: relay hardcoded `protocolVersion: 2025-03-26` and dropped `instructions`, and `prompts/get` without a name returned `{}` instead of the default context prompt. Both transports now delegate `initialize`, `prompts/list`, and `prompts/get` to the same `McpBridge` methods. A non-string `name` on `prompts/get` coerces to undefined and yields the default prompt.

Test suite: 2955 passing plus 1 expected fail, no regressions. Type check clean. Build clean (4.8 MB).

## v2.14.8 -- skill-translator + thinking-toggle reliability pass

Three follow-up fixes against issues that surfaced during live testing of v2.14.7. The skill-translator's ZIP source path now works end-to-end, the dry-run mapping table is broad enough to cover the typical Anthropic stdlib surface, and the v2.14.0 thinking toggle finally reaches the OpenAI-compatible local backends.

- **FIX-29-08-01: skill-translator handles ZIP sources properly.** The translator's SKILL.md offered "ZIP file in vault" as a source option since v2.11.0 but had no concrete extraction path -- the agent was improvising with `evaluate_expression` and hitting three different failure modes (jszip via `dependencies` throws `Cannot read properties of undefined (reading 'bind')`, fflate returns a single empty file because `vault.read()` UTF-8 decoding loses high bytes, and `ctx.vault.readBinary` doesn't survive the postMessage structured-clone bridge). The fix is a new built-in tool `extract_zip` that runs outside the sandbox, with path-traversal and zip-bomb guards plus an optional `strip_root_folder` for Anthropic-style archives. Step 1 of skill-translator's SKILL.md now points at it explicitly and warns against the evaluate_expression path. Nine helper tests pin the guard logic.
- **FIX-29-08-02: dry-run mapping table covers 94 stdlib + npm modules.** The previous 28-entry table missed enough common modules (`importlib`, `dataclasses`, `enum`, `glob`, `fnmatch`, `tomllib`, `bs4`, `pydantic`, `asyncio`, ...) that real Anthropic skills landed in the unmappable branch even when they were importing only stdlib. The table now classifies 27 stdlib mappings, 15 npm mappings, 17 partial mappings with documented limitations, 13 built-in-tool reroutes, and 22 explicitly unmappable entries (`flask`, `fastapi`, `dotenv`, `PIL`, `torch`, ...). `importlib` in particular is now `partial` with the explicit rule "constant module-name maps to a static import plus a dispatch object; runtime-computed names remain unmappable". SKILL.md step 3 also rewrites the abort hand-off so a fallback to `skill-creator` ships the full source path plus the unmappable and partial module lists and tells skill-creator to treat the Python scripts as pseudocode rather than a translation target. 62 new tests pin 53 must-have module classifications plus three structural invariants.
- **FIX-04-03-10: thinking-toggle reaches openai-compat local backends.** Issue #44 (Qwen3 / Gemma3 on oMLX, vLLM, LM Studio, Ollama) reported the per-conversation Thinking off toggle had no effect on `<think>` blocks. Root cause: in `src/api/providers/openai.ts` the thinking-control branch was gated on the OpenRouter discriminator only, so `custom`, `ollama`, and `lmstudio` shapes never reached the disable mechanism the servers expose through their `parameters` object. The fix routes the toggle through all four shapes, with a regression test that pins {provider shape} x {thinking on, off} x {expected request body}.

Bot-compliance pass alongside the fixes: five sentence-case errors and one unnecessary type-assertion in production code (en.ts settings hints + `jwt-decode.ts` claim-structure helper), plus one real duplicate CSS selector (`.checkpoint-marker::before/::after` shared a multi-selector block and `::after` re-declared just the background -- now split into two self-contained blocks). All Review Bot tiers report clean.

Test suite: 2766 passing plus 1 expected fail. Build clean. ESLint clean on changed files.

The runner side of the upcoming Python execution lane (Spec FEAT-29-16, ADR-135) was moved to its own repository at `pssah4/mcp-python-runner-dev`. The plugin keeps a thin backlog row pointing at that repo; no runner code lands in this release.

## v2.14.6 -- security fixes + review-bot compliance

A maintenance patch that closes two Dependabot advisories on a transitive runtime dependency and resolves the four findings the Obsidian Community Plugin Review Bot raised against the v2.14.5 release.

- **`protobufjs` bumped from 8.2.1 to 8.6.3.** The `>=7.5.5` override never reached the 8.x line that `onnxruntime-web` pulls in. The new `>=8.6.0` override covers both advisories: GHSA-wcpc-wj8m-hjx6 (high, unbounded Any expansion in JSON conversion) and GHSA-f38q-mgvj-vph7 (medium, schema-derived names shadowing runtime-significant properties). `npm audit` no longer reports protobufjs.
- **`@smithy/node-http-handler` is now a direct dependency.** The v2.14.5 Bedrock gateway path imports it explicitly but the package was only transitively available via `@aws-sdk/client-bedrock-runtime`. The Review Bot flagged the missing top-level declaration; the dependency is now pinned at `^4.0.0` and resolves to the same 4.7.1 the SDK already used.
- **Two `eslint-disable obsidianmd/ui/sentence-case` directives removed.** The Review Bot blocks any disable of an `obsidianmd/*` rule, including disables with a reason. ## Highlights

### Retrieval Wave 1

The hybrid search pipeline (semantic + lexical + graph, fused with reciprocal rank fusion) got six focused improvements:

- **Opener-chunk excerpts.** Search results now prefer the note's opening chunk for the displayed excerpt instead of whatever chunk happened to rank first. The opener is almost always the most descriptive line, so result lists read like a table of contents rather than a pile of mid-paragraph fragments.
- **Tokenizer folds umlauts and keeps short acronyms.** The lexical arm now folds `ä/ö/ü/ß` so a query for "Ressourcenübersicht" matches "Ressourcenubersicht" and vice versa, and it stops dropping short all-caps tokens, so "KI", "RAG", and "MCP" survive tokenization instead of being filtered as noise.
- **Weighted RRF with a cosine blend.** Fusion now weights the three arms and blends in the raw cosine similarity as a bonus on top of the rank-based score (bonus-only, so a weak semantic hit can never outrank a strong lexical one). Tag-only matches are down-weighted so a note does not float to the top purely because it shares a tag with the query.
- **Typed graph labels from frontmatter.** Graph-arm hits now carry a typed relationship label derived from the frontmatter property that linked them, so the result explains *why* a note is related, not just that it is.
- **Batched reranking with backoff re-arm.** The cross-encoder reranker now processes candidates in batches and re-arms itself after a transient failure instead of staying disabled for the rest of the session.
- **Reranker ONNX runtime fixed.** The reranker model loads again: it now injects the `onnxruntime-web/wasm` backend with `device: 'auto'` and single-threaded WASM (the bundled Electron runtime has no worker-script URL, which broke the threaded build). Confirmed loading in under two seconds.

Two supporting pieces ship with the wave: near-empty stub notes (under 40 characters of real content) are gated out of the semantic index so they stop ranking on title alone, and a light retrieval bench harness landed for regression-checking fusion changes.

### Per-conversation thinking and reasoning effort

The chat model picker gains two controls that apply to the current conversation without touching your saved model settings:

- **Thinking toggle.** A binary On/Off switch. On models that do not support extended thinking it is silently ignored. The state is mirrored on the chat header so you can see it at a glance.
- **Reasoning-effort slider.** A pill slider with the model's own native effort levels. Claude exposes Low / Medium / High / XHigh / Max; GPT-5 and the o-series expose Minimal / Low / Medium / High. The slider is shown only when you pin a specific model (in Auto mode the tier router already picks the model for the task, so effort stays at the model's own default and no effort field is sent). The leftmost stop is always "Auto", which sends nothing and lets the vendor default apply.

Effort reaches every default provider through its native parameter: Anthropic `output_config.effort`, OpenAI and Copilot `reasoning_effort`, OpenRouter `reasoning.effort`, and Amazon Bedrock `reasoning_config`. Each provider validates the level against the model family, so a GPT-only level can never reach a Claude model and vice versa.

### Amazon Bedrock extended thinking

Bedrock now wires extended thinking for the budget-token Claude lineup (Sonnet 4.6, Opus 4.6 and older). Turning thinking on for one of those models sends `reasoning_config { type: 'enabled', budget_tokens: N }` and forces the request temperature to 1, matching the direct Anthropic provider. The effort-capable lineup (Opus 4.7/4.8, Fable) keeps using the effort enum. A capability fix underpins this: only the adaptive-thinking Claude models accept an effort enum, so the budget-token models no longer offer or send one (they previously triggered a Bedrock 400, "thinking.enabled.budget_tokens: Field required").
