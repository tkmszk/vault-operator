# Vault Operator v2.12: Ingest-deep workflow rebuilt, attachment auto-save, slash commands with attachments

The v2.12 release rebuilds the `/ingest-deep` workflow from the ground up and fixes two long-standing issues with how chat attachments interact with skills. The headline change: a PDF you drop into the chat now lands in your attachment folder automatically, and `/ingest-deep` actually walks you through triage, topic selection, and output mode -- no more silent "I'll do it for you" runs.

## Highlights

### Ingest-deep skill rebuilt around explicit user control

The old `/ingest-deep` flow leaned on a naive paragraph-picker inside the `ingest_deep` tool, produced transcript-snippet zettel with source-prefix titles, and skipped the user dialogue whenever stub notes existed in the vault. v2.12 turns the entire skill into a strict seven-step sequence:

1. `ingest_triage` on the source.
2. Triage card with cluster match, related notes from vault / facts / chats, plus the agent's own ingest / spaeter / verwerfen recommendation.
3. `ask_followup_question` for the decision -- the run stops here until you answer.
4. Topic table built from the source, rendered as a numbered markdown table.
5. `ask_followup_question` "which topics?" -- click "Alle" or type a number list.
6. `ask_followup_question` for the output mode (one sense-making note, multiple zettel, source-only, oder doch nicht ingesten).
7. `ingest_deep` with explicit `block_anchors`, then `write_file` for derived notes, then a single `update_frontmatter` on the source that points `Notizen:` at every new note.

Stub notes in the vault no longer trigger a "resume" path -- they are treated as name collisions in step 7 and prompt a follow-up question. Source-mirror frontmatter follows the explicit hygiene rules (gequotete Wikilinks, single `Kategorie`, quoted text values incl. `Zusammenfassung:`, themes pulled from the triage card).

### Triage now searches vault, memory, and chat history

`ingest_triage` no longer returns just a cluster classification. It now runs three parallel searches inside the tool:

- **Vault** -- semantic index top-K (with self-exclusion so the source doesn't match itself)
- **Memory** -- token-overlap over up to 5000 facts (covers any realistic personal fact store)
- **History** -- LIKE-search across `history_chunks` with longest-token probing

The card lists every hit so the user (and the agent's recommendation) can see whether the source is new material, a duplicate, or contradicts something in the vault. Result is no longer externalised (the curated card stays in-context), which also sidesteps the EPERM cleanup warning on iCloud.

### Chat attachments now auto-save to the attachment folder

When you drag a PDF (or DOCX, XLSX) into the chat, the `AttachmentHandler` writes the binary to your Obsidian attachment folder (`attachmentFolderPath` from `.obsidian/app.json`, default `Attachements/`). The `<attached_document>` block gets a `vault_path` attribute pointing at the saved file, so follow-up turns can reach the binary via `move_file` for naming or via `ingest_document` for parsing.

This finally closes the "chat-attachment lives only one turn" gap for binary documents. PPTX/POTX continue to land in `Tools & Settings/Templates` as before.

### Slash commands work with chat attachments

Previously, `/ingest-deep` plus a PDF drop would skip the skill-body expansion (`<explicit_instructions>` was only built on the plain-string path; attachments switched the payload to a `ContentBlock[]` array). The parent agent fell back to `invoke_skill`, the sub-task ran without chat-attachment access, and the parent improvised the workflow. v2.12 expands the slash command BEFORE the attachment-block build, so the skill body reaches the agent whether or not you dropped a PDF.

### Block-anchor matching is fuzzier

`BlockIdSetter` adds four-pass matching: exact substring, whitespace-normalised, punctuation-tolerant, and a longest-contiguous-token-overlap fallback (requires the anchor to cover at least half its tokens). The `ingest_deep` tool also now accepts an explicit `block_anchors` input, so the skill passes the exact source phrases for each take-away. When an anchor fails to match, the tool result lists the unmatched anchors with a clear action ("either re-run with corrected anchors or reference the source without `#^block`").

### Source notes no longer duplicated

`OutputModeGenerator` was rewritten around a single `notesFolder` setting (from `defaultOutputFolder`, default `Inbox/`). It writes Sense-Making / Bibliography / Zettel notes into that one folder; it never copies a source note into `Sources/<basename>.md` again. When the caller passes `existingFile`, the generator sets block-IDs in place via `vault.modify` and preserves the original frontmatter intact.

PDF markdown mirrors now land in the same `notesFolder` (Inbox by default) instead of the PDF's own folder. The binary stays in the attachment folder; only the mirror moves to the inbox.

### Cost-warning footer disabled by default

`costWarnThresholdEur` flips from `0.5` to `0` -- the orange warning style is opt-in now. Users who liked the visible threshold can put a number back via the `update_settings` tool.

## Code-level changes

- `AttachmentHandler.saveExternalBinaryToAttachments()` -- new helper, reads `attachmentFolderPath`, writes the binary, surfaces a vault notice for both success and failure.
- `AgentSidebarView` -- slash-command expansion moved ahead of the block-build so it applies to messages with attachments.
- `BlockIdSetter.findAnchorLine()` -- adds aggressive-normalise (punctuation/quotes stripped) and longest-contiguous-overlap passes.
- `IngestDeepTool` -- accepts `block_anchors` input, returns the resulting anchor-to-block-id map plus an unmatched-anchor warning in the tool result.
- `IngestTriageTool` -- runs vault/memory/history search internally and renders related-context blocks. Output is skip-externalize so the curated card reaches the model verbatim.
- `OutputModeGenerator` -- `OutputFolderConfig` collapses to a single `notesFolder`; new `existingFile` path on `SourceContent` skips the duplicate-note creation and modifies the original frontmatter via `splitFrontmatter` helper.
- `PdfMarkdownMirror` -- accepts a `mirrorFolder` option; mirror lands in the inbox instead of next to the PDF.
- `DeepIngestPipeline` -- `DeepIngestResult` now carries `anchorToBlockId` for the agent to render block-refs.
- `ResultExternalizer` -- `ingest_triage` added to the skip-externalisation set.

## Migration notes

- No data migration. `Sources/` and `Knowledge/<cluster>/` folders that earlier runs created stay where they are; new ingests will not write into them anymore.
- If you had `costWarnThresholdEur` set explicitly, the value carries over; only the new-user default flips.
- The `ingest_deep` tool's `output_mode` parameter still accepts `source-plus-summary` and `source-plus-multi-zettel`, but the `/ingest-deep` skill no longer uses them -- the skill always passes `source-only` and writes derived notes itself for full control over titles, frontmatter, and block-refs.

## v2.12.3 -- Undo-from-here actually rolls back the open note

### What was broken

You ask the agent to clean up a note's frontmatter. The pipeline writes a pre-write checkpoint, the change lands on disk, you click "Undo from here" in the sidebar. The console says `[Checkpoints] restored via vault.modify` and `Restored 1 files` -- but the note in Obsidian still shows the modified state. Next keystroke or auto-save flushes the editor's stale buffer back to disk and silently undoes the restore.

Same mechanism also affected single `edit_file` / `write_file` / `append_to_file` calls against a note that was open in the active editor: the on-disk write succeeded, the read-back proved it, but the visible editor buffer overwrote it within seconds.

### Root cause

`vault.modify()` writes to disk but does not refresh open `MarkdownView` instances. The CodeMirror buffer keeps its pre-change state, and the next save or edit pushes that stale buffer back to disk.

### Fix

Checkpoint restore and the file-write tools now replace the editor buffer directly via `editor.setValue(...)` when the affected note is open in an active leaf, instead of relying on `vault.modify()` alone to propagate. Disk and editor stay in sync, "Undo from here" actually undoes, and an `edit_file` against the currently-open note survives the next save.

If no editor is open on the file, the existing `vault.modify()` path runs unchanged.

## v2.12.4 -- Security maintenance (qs DoS override)

### What changed

Bumps the transitive `qs` dependency from 6.15.1 to 6.15.2 via npm override to close GHSA-q8mj-m7cp-5q26 (CVE-2026-8723). `qs` reaches the project only through `@modelcontextprotocol/sdk` -> `express` -> `body-parser`, so the practical exploit path against the plugin is effectively nil (express uses `qs.parse`, the vulnerable call is `qs.stringify` with `encodeValuesOnly: true` on null/undefined comma-array entries, which the plugin never invokes). The bump keeps the audit baseline clean and closes the Dependabot alert.

No code changes in `src/`. No behaviour changes for users. Full audit details: AUDIT-031.

## v2.12.5 -- DeepSeek deepseek-reasoner unblocked for tool use

### What was broken

Configuring DeepSeek's `deepseek-reasoner` model via a custom OpenAI-compatible provider made any tool-using conversation fail after the first turn with `400 The reasoning_content in the thinking mode must be passed back to the API`. Pure chat ("who are you") worked, but as soon as the agent triggered a tool the follow-up request was rejected and the conversation hung. Reported by DavidHinton-xcpc in issue [#38](https://github.com/pssah4/vault-operator/issues/38).

### Root cause

DeepSeek's reasoner returns both `reasoning_content` (chain-of-thought) and `content` (visible answer or `tool_calls`). The DeepSeek API requires the original `reasoning_content` to be echoed back on the next request that delivers the tool results. The plugin was streaming `reasoning_content` into the UI for display but discarding it before assembling the follow-up request, so the second turn always failed.

### Fix

The internal message model now carries a first-class `thinking` content block. The OpenAI-compatible provider stores reasoning that the model marked as round-trippable, and re-emits it as `reasoning_content` on the assistant message whose `tool_calls` are being resolved. Only the most recent such message is echoed back, so the per-request token cost stays flat regardless of session length -- this matches DeepSeek's multi-round convention of stripping older reasoning.

The passback is opt-in by provider type: `custom`, `ollama`, and `lmstudio` echo, while `openai`, `azure`, `gemini`, and `openrouter` are excluded. OpenAI native and Azure don't expect the field, and OpenRouter has its own server-side reasoning passthrough for Claude Extended Thinking that this change deliberately doesn't touch. If you route DeepSeek through OpenRouter and want it covered, open an issue and we'll extend the allow-list after a regression check against Claude.

Anthropic and Bedrock get a defensive pre-filter so a conversation that started under DeepSeek and is reopened under Claude doesn't crash their strict message conversion. Anthropic's own signed extended-thinking + tool-use round-trip stays out of scope and is tracked separately.

The persisted conversation now carries the captured reasoning text alongside the existing collapsed "Steps" block, so when you reopen an old DeepSeek-reasoner chat you see the chain-of-thought as a collapsed "Reasoning..." bubble above each assistant turn.

## v2.12.6 -- Security patch and provider fixes

### Security maintenance (tmp symlink CVE override)

Bumps the transitive `tmp` dependency to `>= 0.2.6` via npm override to close GHSA-ph9p-34f9-6g65 (CVE-2026-44705), an arbitrary-file-write-via-symlink issue. `tmp` reaches the project only through `exceljs` (the XLSX export path). The bundled tree now resolves `tmp` 0.2.7 and `npm audit` is clean again. Full audit details: AUDIT-032.

### Clearer model-ID entry when a provider has no /v1/models

Adding a custom OpenAI-compatible provider whose endpoint does not implement `/v1/models` returned an empty model list with no obvious next step. The "no models found" and "server unreachable" hints now tell you to type the model ID straight into the Model ID field and save, so a missing discovery route no longer blocks setup. Reported as issue [#40](https://github.com/pssah4/vault-operator/issues/40) (FIX-04-03-40).

### Reliability hardening (checkpoints and edit steering)

Regression coverage was added for two edit and checkpoint paths so they cannot silently regress. A new-file-only checkpoint snapshot now provably commits a real revision, so undo can delete the file again instead of leaking it. And `edit_file` keeps steering large rewrites toward `write_file` at the 1000-character threshold. There is no behaviour change here -- the fixes already shipped; this release pins them with tests.

### Dependency note

The review bot may list `uuid` and `tmp` as reachable through `exceljs`. Both are forced past their vulnerable ranges by npm `overrides` (uuid 14.x, tmp 0.2.7) and `npm audit` reports zero vulnerabilities. The warnings reflect the ranges `exceljs` declares, not the versions actually bundled.

## v2.12.8 -- Code-review wave and ingest skill discipline

Two independent strands ship together. The headline is a seven-fix correctness wave that came out of a focused multi-angle code-review of the provider layer, the vault edit tools, and the attachment pipeline. The second strand is a property-discipline rule for the three ingest skills so that `Themen` and `Konzepte` stop drifting into each other.

### Vision input now reaches OpenAI-shape providers

Dropping a PNG or JPEG into the chat used to leave OpenAI, GitHub Copilot, and the Kilo Gateway looking at text only. Their `convertMessages` only matched `text` and `tool_result` user blocks; image blocks fell through. Anthropic and Bedrock already handled the same blocks correctly. The three OpenAI-shape providers now emit the canonical `{type:'image_url', image_url:{url:'data:<media-type>;base64,...'}}` content-array on user messages whenever an image block is present, while pure-text messages continue to send `content` as a string for backwards compatibility. gpt-4o, Gemini via OpenAI-compat, OpenRouter vision models, and the Kilo Gateway Claude tiers can now describe what you dropped.

### EditFileTool fuzzy-match preserves untouched formatting

When `edit_file` failed an exact `old_str` match it would fall back to a whitespace-lenient comparison. The old fallback then collapsed multi-space runs, normalised CRLF to LF, and `.trim()`-stripped the whole file before writing the result back. Anywhere outside the matched edit -- tab-indented code fences, aligned tables, YAML frontmatter indentation -- got silently rewritten. The new behaviour normalises only for the search step, maps the matched range back to the original file via an index walk, and splices `new_str` into exactly that range. Multi-match ambiguity is escalated as a hard error instead of silently picking the first hit, so the agent has to add disambiguating context. Four regression tests pin the new contract.

### AttachmentHandler stops silently swapping content

When two different files arrived in the chat with the same name, the attachment auto-save returned the existing vault path without writing the new bytes -- but the new parsed text was still pushed into chat history. Every follow-up `read_document` then served the old bytes under the new name. The new collision path compares bytes first: identical bytes reuse the existing path with a "duplicate" notice; differing bytes get the next free `<base>-N.<ext>` slot, with the suffix landing before the final dot so `archive.tar.gz` becomes `archive.tar-2.gz` and stays openable by extension-based tools.

### Kilo Gateway gains tool-call streaming parity

Three Kilo Gateway gaps closed in one pass.

The post-stream tool-call flush that `openai.ts` and `github-copilot.ts` already had for the BUG-013 / FEATURE-0409 pattern (upstreams that emit `finish_reason="stop"` or `"length"` while still streaming tool-call deltas) was missing on the Kilo Gateway. Tool calls were silently dropped when an upstream Groq or OpenRouter or Claude tier did not end the turn with `tool_calls`. The new shared `flushToolCallAccumulators` helper in `utils/toolCallFlush.ts` is now the single canonical implementation across all three OpenAI-shape providers.

The Kilo Gateway also strict-typechecked `delta.content` as a string and dropped the Claude-shim array form `[{type:'text', text:'...'}]`. GitHub Copilot already normalised this. The helper now lives in `utils/openAiContent.ts` and both providers use it.

### Truncated tool inputs get the right recovery message

When a model hits `max_tokens` mid-`write_file` and the streaming JSON ends mid-string, the resulting tool error told the model "arguments were truncated or malformed" -- the same wording it used for a syntactically wrong payload. Models that hit this then retried with the same long content, hit the cap again, and burned through `consecutiveMistakeLimit`. The post-stream flush in OpenAI, Copilot, and Kilo Gateway now tracks `finish_reason` across the stream and passes `wasMaxTokens=true` to `truncatedToolInputError` when the cut happened at the budget. The error then carries the actionable "Response hit the max output token limit -- split write_file with the document header and append_to_file for the rest" guidance Anthropic already produced. The chatgpt-oauth Responses-API surface uses a different `incomplete` status field and is tracked as a follow-up.

### estimatePromptTokens now counts tool schemas

The token budget helper that decides how aggressively to clamp `max_tokens` in `resolveOutputBudget` summed only the system prompt and the message content. A vault-operator active-mode request ships sixty tool definitions, which add roughly twenty thousand tokens that OpenAI, OpenRouter, and Gemini-via-OpenAI count against the input window. The helper missed those, the clamp came in too generous, and the provider answered with a 400 "context_length_exceeded" before any streaming. `estimatePromptTokens` now takes an optional `tools` array; all five providers pass their tool list through.

### Themen / Konzepte property discipline in the ingest skills

The `knowledge-ingest`, `ingest`, and `ingest-deep` skills now share an explicit rule for the two wikilink properties: `Themen` holds only wikilinks to notes with `Kategorie: Thema` (broad hub notes like `[[Agentic AI]]`), `Konzepte` holds only wikilinks to notes with `Kategorie: Konzept` (specific concept notes like `[[AI Agents]]`). The two lists never mix. Both properties are always YAML lists -- either block form (`Themen:\n  - "[[X]]"`) or flow form (`Themen: ["[[X]]"]`), never the comma-separated string `Themen: [[X]], [[Y]]` that Obsidian does not parse as a list. Even a single value stays a one-element list so later appends do not have to switch the type. The rule also fixes inconsistencies in the proposal snippets where the skills used to show one form in one place and a different form three sections down.

### Hub-note auto-update -- a design memo, not a feature

A live question during the release prep pointed at a real gap: the hub notes that organise the vault (`Themen/Agentic AI.md`, `Konzepte/AI Agents.md`) do not update themselves when a new note references them. Obsidian's native backlinks panel is the only user-visible surface, and it lives in the sidebar, not in the hub note body. The `TopHubBlockGenerator` and `ImplicitConnectionService` run only in the system-prompt and database layers. This release does not add an auto-update. The new design memo `RESEARCH-37` walks four options -- companion `.base` per hub, skill-edit of foreign notes, a periodic plugin job, and a single hub-index note -- and recommends a skill-only Option A plus D combination at roughly one to two days of work. Tracked as `IMP-19-32-01` in EPIC-19 Candidates, pending the user's option choice and the open scoping questions in the memo.

### Security

Independent delta audit `AUDIT-034` walked all ten changed source files and concluded: zero Critical, zero High, zero Medium, zero Low, one Info finding. The Info entry notes a time-of-check-time-of-use window between `getAbstractFileByPath` and `createBinary` in the new collision-rename path; Obsidian runs a single JavaScript process and the only writer is the sidebar drop handler, so the worst case is a local-vault inconsistency rather than a remote attack. `npm audit` reports zero vulnerabilities across one thousand and six packages, unchanged from the AUDIT-033 baseline. All seven fixes are net-defensive: silent failure modes were replaced either by correct behaviour or by a hard error that surfaces to the agent.
