# Vault Operator v2.12: Ingest-deep workflow rebuilt, attachment auto-save, slash commands with attachments

The v2.12 release rebuilds the `/ingest-deep` workflow from the ground up and fixes two long-standing issues with how chat attachments interact with skills. The headline change: a PDF you drop into the chat now lands in your attachment folder automatically, and `/ingest-deep` actually walks you through triage, topic selection, and output mode -- no more silent "I'll do it for you" runs.

## Highlights

### Ingest-deep skill rebuilt around explicit user control

The old `/ingest-deep` flow leaned on a naive paragraph-picker inside the `ingest_deep` tool, produced transcript-snippet zettel with source-prefix titles, and skipped the user dialogue whenever stub notes existed in the vault. v2.12 turns the entire skill into a strict seven-step sequence:

1. `ingest_triage` on the source.
2. Triage card with cluster match, related notes from vault / facts / chats, plus the agent's own ingest / spaeter / verwerfen recommendation.
3. `ask_followup_question` for the decision -- the run stops here until you answer.
4. Topic table built from the source, rendered as a numbered markdown table.
5. `ask_followup_question` "which topics?" -- click "Alle" or type a number list.
6. `ask_followup_question` for the output mode (one sense-making note, multiple zettel, source-only, oder doch nicht ingesten).
7. `ingest_deep` with explicit `block_anchors`, then `write_file` for derived notes, then a single `update_frontmatter` on the source that points `Notizen:` at every new note.

Stub notes in the vault no longer trigger a "resume" path -- they are treated as name collisions in step 7 and prompt a follow-up question. Source-mirror frontmatter follows the explicit hygiene rules (gequotete Wikilinks, single `Kategorie`, quoted text values incl. `Zusammenfassung:`, themes pulled from the triage card).

### Triage now searches vault, memory, and chat history

`ingest_triage` no longer returns just a cluster classification. It now runs three parallel searches inside the tool:

- **Vault** -- semantic index top-K (with self-exclusion so the source doesn't match itself)
- **Memory** -- token-overlap over up to 5000 facts (covers any realistic personal fact store)
- **History** -- LIKE-search across `history_chunks` with longest-token probing

The card lists every hit so the user (and the agent's recommendation) can see whether the source is new material, a duplicate, or contradicts something in the vault. Result is no longer externalised (the curated card stays in-context), which also sidesteps the EPERM cleanup warning on iCloud.

### Chat attachments now auto-save to the attachment folder

When you drag a PDF (or DOCX, XLSX) into the chat, the `AttachmentHandler` writes the binary to your Obsidian attachment folder (`attachmentFolderPath` from `.obsidian/app.json`, default `Attachements/`). The `<attached_document>` block gets a `vault_path` attribute pointing at the saved file, so follow-up turns can reach the binary via `move_file` for naming or via `ingest_document` for parsing.

This finally closes the "chat-attachment lives only one turn" gap for binary documents. PPTX/POTX continue to land in `Tools & Settings/Templates` as before.

### Slash commands work with chat attachments

Previously, `/ingest-deep` plus a PDF drop would skip the skill-body expansion (`<explicit_instructions>` was only built on the plain-string path; attachments switched the payload to a `ContentBlock[]` array). The parent agent fell back to `invoke_skill`, the sub-task ran without chat-attachment access, and the parent improvised the workflow. v2.12 expands the slash command BEFORE the attachment-block build, so the skill body reaches the agent whether or not you dropped a PDF.

### Block-anchor matching is fuzzier

`BlockIdSetter` adds four-pass matching: exact substring, whitespace-normalised, punctuation-tolerant, and a longest-contiguous-token-overlap fallback (requires the anchor to cover at least half its tokens). The `ingest_deep` tool also now accepts an explicit `block_anchors` input, so the skill passes the exact source phrases for each take-away. When an anchor fails to match, the tool result lists the unmatched anchors with a clear action ("either re-run with corrected anchors or reference the source without `#^block`").

### Source notes no longer duplicated

`OutputModeGenerator` was rewritten around a single `notesFolder` setting (from `defaultOutputFolder`, default `Inbox/`). It writes Sense-Making / Bibliography / Zettel notes into that one folder; it never copies a source note into `Sources/<basename>.md` again. When the caller passes `existingFile`, the generator sets block-IDs in place via `vault.modify` and preserves the original frontmatter intact.

PDF markdown mirrors now land in the same `notesFolder` (Inbox by default) instead of the PDF's own folder. The binary stays in the attachment folder; only the mirror moves to the inbox.

### Cost-warning footer disabled by default

`costWarnThresholdEur` flips from `0.5` to `0` -- the orange warning style is opt-in now. Users who liked the visible threshold can put a number back via the `update_settings` tool.

## Code-level changes

- `AttachmentHandler.saveExternalBinaryToAttachments()` -- new helper, reads `attachmentFolderPath`, writes the binary, surfaces a vault notice for both success and failure.
- `AgentSidebarView` -- slash-command expansion moved ahead of the block-build so it applies to messages with attachments.
- `BlockIdSetter.findAnchorLine()` -- adds aggressive-normalise (punctuation/quotes stripped) and longest-contiguous-overlap passes.
- `IngestDeepTool` -- accepts `block_anchors` input, returns the resulting anchor-to-block-id map plus an unmatched-anchor warning in the tool result.
- `IngestTriageTool` -- runs vault/memory/history search internally and renders related-context blocks. Output is skip-externalize so the curated card reaches the model verbatim.
- `OutputModeGenerator` -- `OutputFolderConfig` collapses to a single `notesFolder`; new `existingFile` path on `SourceContent` skips the duplicate-note creation and modifies the original frontmatter via `splitFrontmatter` helper.
- `PdfMarkdownMirror` -- accepts a `mirrorFolder` option; mirror lands in the inbox instead of next to the PDF.
- `DeepIngestPipeline` -- `DeepIngestResult` now carries `anchorToBlockId` for the agent to render block-refs.
- `ResultExternalizer` -- `ingest_triage` added to the skip-externalisation set.

## Migration notes

- No data migration. `Sources/` and `Knowledge/<cluster>/` folders that earlier runs created stay where they are; new ingests will not write into them anymore.
- If you had `costWarnThresholdEur` set explicitly, the value carries over; only the new-user default flips.
- The `ingest_deep` tool's `output_mode` parameter still accepts `source-plus-summary` and `source-plus-multi-zettel`, but the `/ingest-deep` skill no longer uses them -- the skill always passes `source-only` and writes derived notes itself for full control over titles, frontmatter, and block-refs.

## v2.12.3 -- Undo-from-here actually rolls back the open note

### What was broken

You ask the agent to clean up a note's frontmatter. The pipeline writes a pre-write checkpoint, the change lands on disk, you click "Undo from here" in the sidebar. The console says `[Checkpoints] restored via vault.modify` and `Restored 1 files` -- but the note in Obsidian still shows the modified state. Next keystroke or auto-save flushes the editor's stale buffer back to disk and silently undoes the restore.

Same mechanism also affected single `edit_file` / `write_file` / `append_to_file` calls against a note that was open in the active editor: the on-disk write succeeded, the read-back proved it, but the visible editor buffer overwrote it within seconds.

### Root cause

`vault.modify()` writes to disk but does not refresh open `MarkdownView` instances. The CodeMirror buffer keeps its pre-change state, and the next save or edit pushes that stale buffer back to disk.

### Fix

Checkpoint restore and the file-write tools now replace the editor buffer directly via `editor.setValue(...)` when the affected note is open in an active leaf, instead of relying on `vault.modify()` alone to propagate. Disk and editor stay in sync, "Undo from here" actually undoes, and an `edit_file` against the currently-open note survives the next save.

If no editor is open on the file, the existing `vault.modify()` path runs unchanged.
