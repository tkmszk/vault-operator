The v2.13 release hardens the contract between the experimental Stigmergy advisor layer and Vault Operator's own selectors (recipes, FastPath, the memory composer). The headline shift is in how recipe promotion works. Until v2.12 a workflow had to repeat itself three times before the plugin condensed it into a learned recipe. v2.13 adds a second path: when the advisor pins a capability sequence and the agent follows it to completion, that single run is enough evidence to persist a recipe candidate. The existing three-organic-similar path stays in place for users who run without the advisor.

## Highlights

### Single-run recipe promotion from advisor-pinned sequences

The plugin already had two parallel mechanisms for turning recurring tool workflows into recipes:

- The static recipe catalogue in `bundled-skills/` (curated by hand).
- The semantic promotion path (ADR-058): after every successful task the promotion service searched the episode store for semantically similar predecessors and, once three landed in the same neighborhood, it called the helper model and saved the resulting recipe to the store.

The semantic path works but it takes weeks of organic use before a pattern crosses the three-occurrence threshold. v2.13 adds a third path. When the advisor returns a `sequence` decision for the turn (a pinned ordered list of capabilities), the plugin pre-activates any deferred tools on that path before the model sees its prompt. If the agent then follows the pinned path to a clean `attempt_completion`, the promotion service treats the single successful run as evidence and writes a learned recipe immediately. The recipe is tagged `provenance: stigmergy-shortcut` and gets a deterministic id derived from a hash of the pinned path, so re-runs of the same path do not fan out into multiple near-duplicates.

The new shortcut goes through every existing gate. `mastery.learnedEnabled` still has to be on. The fifty-recipe cap (`MAX_LEARNED_RECIPES`) still applies. The trigger-token dedup against existing learned recipes still applies. The shortcut also enforces two extra checks: the executed `toolSequence` must contain the pinned path as a contiguous subsequence, and the run must end in `attempt_completion`. Partial runs and runs that drifted away from the pin do not promote. With the advisor turned off or the daemon down, the snapshot is simply absent and the promotion service falls through to the unchanged ADR-058 path; nothing about the old flow changes for users who do not run the advisor.

### Precedence rule: VO selectors win, advisor advises

The advisor and the recipe matcher used to fire independently. When both had something to say about the same turn, the model saw two competing instruction blocks: the recipe section in the system-prompt prefix plus the advisor's guidance text at the end of the user message. The model had to decide which to follow.

v2.13 makes the precedence explicit. When a recipe matches and FastPath actually executes it, the advisor's guidance text is suppressed for that turn. The advisor's path is still used for pre-activation of deferred tools, because that is mechanical and does not compete with the recipe. The cached prefix stays byte-stable -- the recipe section never moves -- and the guidance text only ever lives at the user-message tail. No double-hint, no token waste, and the recipe wins cleanly when it has a confident match.

A second piece keeps the advisor's substrate from learning the wrong lessons. Tool dispatches that come from FastPath now carry a `source: 'fastpath'` tag through the pipeline, and capability-lifecycle events fire only for model-initiated dispatches. The advisor learns from what the model picked, not from what the recipe-driven batch executor did on its behalf.

### Episode recording finally captures every run

The episode pipeline had three quiet failure modes that v2.13 closes.

- **FastPath runs were invisible.** FastPath called the tool execution pipeline directly, but the repetition detector only recorded calls that came from the main loop. The detector therefore reported `toolSequence.length < 2` for FastPath turns and the episode extractor dropped them. The plugin's most valuable training data, recipe-driven success runs, was being silently discarded. v2.13 routes FastPath dispatches through a new `recordForEpisodeOnly` path so the episode captures the full chronological sequence (`search_files -> read_file -> write_file -> attempt_completion`) without affecting the repetition-detection sliding window.
- **Iteration-cap and abort runs vanished from telemetry.** The `onEpisodeData` callback used to fire only from the success branch. Runs that hit the iteration cap or were aborted by the user produced no episode at all, which made them invisible to bug diagnostics. The callback now fires from the `finally` block with the full payload, including a `success` flag that derives from the advisor's outcome grade plus an internal mistake counter plus whether `attempt_completion` was ever called. Failed episodes go to the database for telemetry but do not get added to the semantic index, so the memory retriever stops surfacing trial-and-error fragments in future searches.
- **The advisor decision was never persisted.** The decision was in scope inside `AgentTask.run()` but never made it to the episode record, so the promotion service could not tell a stigmergy-guided success from an organic one. The new `stigmergy_json` column on the `episodes` table holds the per-turn snapshot ({ mode, pinnedPath, guidanceTextSuppressed, recipeWinner }) and the promotion service reads it directly. The snapshot carries only capability ids (skill slugs, MCP server/name pairs, plain tool names) plus booleans and the winning recipe id, never any user text.

### Episode database schema bump (v4 â†’ v5)

The `episodes` table gets one additive column: `stigmergy_json TEXT` with a NULL default for legacy rows. The bump runs idempotently inside `MemoryDB.open()` before any concurrent writer wakes up. SELECT defends against malformed JSON with a try/catch and runs a runtime shape guard so a manually edited DB or a future writer that drifts from the contract cannot feed garbage to the promotion gates. The eviction policy switches from `ORDER BY created_at` to `ORDER BY rowid` so a clock skew (multi-device sync, manual datetime reset) cannot evict the wrong row.

### Memory robustness

Two smaller robustness pieces ship alongside.

- **Topic-slug normalization on the fact store.** Memory v2 used to write the topic exactly as the extractor produced it, which meant the same concept appeared as `Plan Mode`, `planMode`, `plan-mode`, and ` Plan-Mode ` across different extractions. Search for one of those variants missed the others. v2.13 normalises every topic to a lowercase hyphen-separated slug on insert (`plan-mode`), and the `normalizeTopics` helper dedupes the array. New facts only -- existing rows are unaffected.
- **Skill discovery has a 1500ms ceiling.** A hanging user-skill folder used to block the AgentTask while `skillsManager.discoverSkills()` waited on a slow `readdir`. v2.13 wraps the call in a `withTimeout` helper and continues without the user-skill enumeration on timeout. Self-authored skills remain available; the agent just doesn't see the user folder this turn.

## Security audit

The branch went through AUDIT-036 (per-item delta on top of AUDIT-035 from June 3rd) before merge. Verdict GREEN, release-readiness GREEN. Eight findings initial (1 Medium, 5 Low, 2 Info), all six actionable findings resolved in the fix-loop:

- **hono** was bumped from `>=4.12.18` to `>=4.12.21` in the `overrides` block to close four published advisories (IPv6 IP-restriction bypass, cookie sameSite sanitisation, JWT middleware auth-scheme, app.mount route confusion). `npm audit` now reports 0 vulnerabilities across 1010 packages.
- The Stigmergy adapter's `__testHooks` export is now gated behind a `NODE_ENV` check, so the production bundle ships `undefined` for the test mutators.
- The `stigmergy_json` SELECT path runs a runtime shape guard (`isEpisodeStigmergySnapshot`) so malformed snapshots are rejected before they reach the promotion gates.
- The pipeline's `source` parameter now produces a `[Substrate-Skip]` debug-log line when a non-`model` dispatch happens on an enabled advisor turn, so a misuse from a future caller is grep-visible in session transcripts.
- The recipe-generator prompt in `promoteFromStigmergyPath` wraps the user message in explicit `<user_message>` markers, strips ASCII control characters, and caps the input at 500 characters; the meta-prompt instructs the model to treat the marker block as data, not instructions.
- The `appendGuidanceText` helper's input parameter is now `ReadonlyArray<UserContentBlock>`, which surfaces the no-mutation contract at the type level.

Two Info findings (`withTimeout` does not abort the inner promise, MemoryDB v5 migration runs without explicit WriterLock) are accepted by design and documented in the audit report at `_devprocess/analysis/AUDIT-036-epic32-stigmergy-hardening-2026-06-07.md`.
