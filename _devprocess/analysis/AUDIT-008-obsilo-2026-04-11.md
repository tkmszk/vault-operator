### P3: Consider (Low + Info)

#### L-1 NEU: Device-Auth Polling ohne Max-Attempts (CWE-400)

| Field | Value |
|-------|-------|
| **Severity** | Low |
| **CWE** | CWE-400 (Resource Exhaustion) |
| **Location** | `src/core/security/KiloAuthService.ts:170-205`, `src/core/security/GitHubCopilotAuthService.ts:205` |
| **Status** | Confirmed (by design) |

**Finding:** `pollForSession()` und das GitHub Copilot Pendant nutzen `while(true)` ohne
MAX_ATTEMPTS-Limit. Abbruch nur durch `AbortSignal` (User schliesst Modal).

**Bewertung:** Standard-OAuth-Device-Flow-Verhalten. Der User kontrolliert den Abbruch ueber
die UI. Kein externer Angriffsvektor -- der Polling-Endpoint ist der eigene Auth-Server.
Maximaler Impact: Obsidian pollt im Hintergrund weiter falls der User das Modal nicht schliesst.

**Empfehlung (optional):** MAX_ATTEMPTS = 120 (= 10 Minuten bei 5s Intervall) als Sicherheitsnetz.

## Positive Findings

Die Codebase zeigt weiterhin eine reife Sicherheitsarchitektur:

- **XSS-Praevention:** 0 innerHTML-Nutzungen. Konsequent Obsidian DOM API.
- **Sandbox-Isolation:** Dreistufig (AST-Validation -> vm.createContext -> OS-Prozess-Isolation).
- **Path-Traversal-Schutz:** 3-Layer-Checks in GlobalFileService und recipeValidator.
- **SSRF-Schutz:** 2-Phasen DNS-Rebinding-Check in WebFetchTool.
- **Tool-Governance:** Granulares Approval-System. Self-Modification IMMER manuell.
- **Excessive Agency Controls:** consecutiveMistakeLimit, maxIterations, rateLimitMs, maxSubtaskDepth, ToolRepetitionDetector.
- **Credential-Handling:** SafeStorageService + Bearer-Auth + Constant-Time Vergleich.
- **Error-Sanitization:** Stack-Traces an API-Grenzen gestrippt.
- **Zero Hardcoded Paths:** `vault.configDir` ueberall.
- **Review-Bot Compliance:** Alle eslint-disable fuer obsidianmd-Regeln entfernt (diese Session).
- **i18n-Migration VaultHealthRepairModal:** Eliminiert hardcoded Strings in Code (Clean Architecture).
