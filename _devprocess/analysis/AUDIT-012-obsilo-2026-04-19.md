## 3 Findings Detail

### M-1: HTML-comment metadata block has no length limit

- **Severity:** Medium
- **CWE:** CWE-1333 (Inefficient Regular Expression Complexity) + CWE-400 adjacent
- **Location:** [SelfAuthoredSkillLoader.ts:845](../../../src/core/skills/SelfAuthoredSkillLoader.ts#L845)
- **Status:** Accepted (design trade-off, deferred to v2.7)
- **Risk.** The EPIC-22 beta feedback added a fallback parser for
  `<!-- Metadata ... -->` blocks that some real-world Anthropic skills
  ship. The regex `/<!--\s*(?:Metadata|metadata|SKILL|skill)\s*\n([\s\S]*?)\n-->/`
  is non-greedy but unbounded. A crafted 10 MB SKILL.md with a fake
  metadata comment would drive `parseFrontmatter()` (line-by-line loop)
  into O(n) string processing on an attacker-controlled payload.
- **Practical exposure.** Low. SKILL.md files come from the user's own
  import (zip or folder picker) and the whole zip is already capped at
  100 MB of uncompressed content. No remote ingestion path exists.
- **Evidence.**
```typescript
const html = content.match(/<!--\s*(?:Metadata|metadata|SKILL|skill)\s*\n([\s\S]*?)\n-->/);
if (html) {
    frontmatter = html[1];                  // no length cap
    body = content.replace(html[0], '').trim();
}
```
- **Remediation (v2.7).** Add `if (html[1].length > 65_536) return null;`
  before the parse, and cap `parseFrontmatter()` at e.g. 500 lines. Both
  are one-line changes.
- **Decision.** Accept for v2.6.0. Documented in the v2.7 backlog.

### M-2: TOCTOU between exists() check and writeBinary() in SkillPackageImporter

- **Severity:** Medium
- **CWE:** CWE-367 (Time-of-check Time-of-use)
- **Location:** [SkillPackageImporter.ts:156-174](../../../src/core/skills/SkillPackageImporter.ts#L156-L174)
- **Status:** Mitigated (UI gate + overwrite flag)
- **Risk.** The importer checks `adapter.exists(destFolder)` then
  `mkdir` + `writeBinary`. Obsidian's adapter does not expose atomic
  file creation, so concurrent imports could race. An attacker who can
  drive two simultaneous imports could in theory write into a folder
  they did not intend to create.
- **Practical exposure.** Low. Imports happen through the Settings
  UI file picker; a single user-initiated action per click. No API
  endpoint, no background trigger. Duplicate detection also gates on
  user confirmation before overwriting.
- **Remediation.** Accept. Obsidian Plugin API does not support atomic
  file create; Node `fs.promises.mkdir` with `{ recursive: false }`
  would throw on an existing dir but the codebase already relies on
  the adapter. No realistic attack path in the single-user Electron
  runtime.

### L-1: JSZip uses a private `_data.uncompressedSize` field

- **Severity:** Low
- **CWE:** CWE-20 (Improper Input Validation) -- brittleness
- **Location:** [SkillPackageImporter.ts:215-219](../../../src/core/skills/SkillPackageImporter.ts#L215-L219)
- **Status:** Mitigated (conservative fallback)
- **Risk.** `getUncompressedSize()` reaches into JSZip's private
  `_data` field. If a future JSZip release renames this field, the
  zip-bomb guard silently falls back to `compressedSize`, which for a
  zip bomb is much smaller than the real payload.
- **Mitigation already in place.** The fallback is conservative for
  defense in depth; JSZip is pinned via `package.json` so a silent
  API break requires an explicit dependency bump.
- **Remediation.** Add a unit test that asserts the private field is
  populated on a known small zip. If JSZip ever removes it, the build
  breaks instead of silently degrading.

### L-2: Bridge request circuit-breaker state shared across skill invocations

- **Severity:** Low
- **CWE:** CWE-362 (Race Condition)
- **Location:** [SandboxBridge.ts:170-210](../../../src/core/sandbox/SandboxBridge.ts#L170-L210)
- **Status:** Mitigated (30 s cooldown design)
- **Risk.** BUG-027 replaced the one-shot breaker with a 30 s
  cooldown. A malicious skill could intentionally fail 20 bridge
  calls to lock a different skill out of the bridge for 30 s.
- **Practical exposure.** Very low. Skills are user-installed and
  reviewed, not code delivered over the wire. A skill that wants to
  DoS the bridge has easier vectors (infinite loops, rate-limit
  burns).

### L-3: Symlink handling in SkillFolderImporter

- **Severity:** Low
- **CWE:** CWE-59 (Improper Link Resolution)
- **Location:** [SkillFolderImporter.ts:142-148](../../../src/core/skills/SkillFolderImporter.ts#L142-L148)
- **Status:** Mitigated (comment explicit, walk() does not follow)
- **Risk.** Node `fs.promises.readdir({ withFileTypes: true })` returns
  the symlink's own dirent, and `entry.isFile()` is false on symlinks
  by default. The comment explicitly says "Symlinks are ignored
  deliberately". Safe.

### L-4: HTML file input accept attribute is client-side only

- **Severity:** Low
- **CWE:** CWE-345 (Insufficient Verification of Data Authenticity)
- **Location:** [SkillsTab.ts:725-727](../../../src/ui/settings/SkillsTab.ts#L725-L727)
- **Status:** Mitigated
- **Risk.** `fileInput.accept = '.md,.txt,.zip,.skill'` is a hint for
  the native picker only. A user can pass any file.
- **Mitigation.** `SkillImportRouter.detectSourceFromFile()` dispatches
  by extension, then the sub-importers validate content:
  JSZip rejects non-zip bytes, markdown-import accepts any text but
  the whole flow is single-user-initiated.

### L-5: Unused eslint-disable directives produced warnings before beta-8

- **Severity:** Low
- **CWE:** N/A (code hygiene)
- **Status:** Resolved 2026-04-18 during BUG-028 fix
- **Note.** Tracked here only for audit completeness.

### I-1 to I-6: Positive findings (already implemented)

- **I-1.** `SandboxBridge.hasPollutionKeys()` prevents prototype
  pollution in `requestUrl` options (comprehensive recursive check).
- **I-2.** Phase-0 path-traversal rejection in
  `SkillPackageImporter` runs BEFORE slug detection, so a crafted
  root dir cannot drive the import target.
- **I-3.** `validateVaultPath` in SandboxBridge explicitly blocks
  writes to `configDir` (the `.obsidian/` folder). Shai Hulud audit
  L-2 mitigation.
- **I-4.** `ReadFileTool.looksLikeExternalisedTmpPath` is a strict
  pattern check with `..` + null-byte rejection. The retry only
  succeeds when the prefixed path ALSO exists on disk, so it cannot
  be used to probe arbitrary filesystem paths.
- **I-5.** `safeRegex()` guards all user-supplied skill trigger
  patterns against ReDoS (pre-existing, reaffirmed in this audit).
- **I-6.** `ResultExternalizer` cleanup retry + orphan sweeper
  provide belt-and-braces cleanup so failed unlinks on iCloud
  vaults never leak data to the wrong place.
