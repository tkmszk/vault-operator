## Findings

### H-1: Path-Traversal in IngestTriageTool source_uri (Confirmed)

**Severity:** High
**CWE:** CWE-22 Path Traversal
**Location:** `src/core/tools/vault/IngestTriageTool.ts:90-95`

**Code:**
```typescript
if (source_uri.startsWith('vault://')) {
    const path = source_uri.slice('vault://'.length);
    const file = this.plugin.app.vault.getAbstractFileByPath(path);
```

**Risk:** Der Tool-Input `source_uri` wird vom LLM-Agent geliefert. Strings wie `vault://../../OtherVault/secret.md` oder `vault://./../../../tmp/x` werden ohne Pfad-Normalisierung an `getAbstractFileByPath` weitergeben. Obsidians Vault-API ist primaer relativ zum Vault-Root, was direkten FS-Escape vermutlich blockiert. Aber:
- Auf Windows koennte `vault://..\..\file` Pfad-Separator-Inkonsistenzen ausnutzen.
- `vault://..%2F..` wird URL-encoded und koennte Decoding-Inconsistencies treffen.
- Selbst wenn Obsidian Path-Escape blockiert, persistiert der String (`source_uri`) unsanitisiert in `ingest_triage_log` und in Triage-Karten-Output.

**Remediation:** Source-URI-Validator vor jedem Lookup:
```typescript
function validateVaultPath(rawPath: string): string | null {
    const normalized = rawPath.replace(/\\/g, '/').replace(/^\/+/, '');
    if (normalized.includes('..') || normalized.includes('\0')) return null;
    return normalized;
}
```
In execute(): `validateVaultPath(path)` vor `getAbstractFileByPath`. Bei null: error.

**Status:** Confirmed (Fix angewandt in dieser Session, siehe Fix-Loop unten).

### M-1: Prototype-Pollution-Risiko in FrontmatterWriter (Confirmed)

**Severity:** Medium
**CWE:** CWE-1321 Prototype Pollution
**Location:** `src/core/ingest/FrontmatterWriter.ts:88-100`

**Code:**
```typescript
for (const [name, field] of Object.entries(patch)) {
    const exists = name in fm && fm[name] !== null && fm[name] !== undefined;
    if (!exists) {
        fm[name] = field.value as never;
        ...
```

**Risk:** Property-Namen kommen vom Patch-Object und werden direkt als Object-Key gesetzt. Wenn ein LLM-Output (zB im Auto-Summary-Pfad) einen Property-Namen wie `__proto__`, `constructor`, `prototype` liefert, koennte das Prototype-Chain-Pollution ausloesen. Zwar ist `fm` ein internes Object aus processFrontMatter und der direkte Prototype-Pollution-Vector ist limitiert, aber best-practice ist Property-Name-Filter.

**Remediation:** Reject von Property-Namen aus der Prototype-Chain:
```typescript
const FORBIDDEN_PROPS = new Set(['__proto__', 'constructor', 'prototype']);
for (const [name, field] of Object.entries(patch)) {
    if (FORBIDDEN_PROPS.has(name)) {
        console.warn(`[FrontmatterWriter] Skipping forbidden property name: ${name}`);
        continue;
    }
    ...
```

**Status:** Confirmed (Fix angewandt).

### M-2: Top-Hub-Block embeddet Note-Summaries in System-Prompt ohne User-Hinweis (Deferred)

**Severity:** Medium
**CWE:** CWE-359 Privacy Violation / OWASP LLM06 Sensitive Info Disclosure
**Location:** `src/core/memory/TopHubBlockGenerator.ts:135-147`

**Code:**
```typescript
private renderBlock(hubs: HubEntry[]): string {
    ...
    for (const m of members) {
        const summary = m.summary ?? '(keine Summary)';
        lines.push(`- [[${filename}]] -- ${summary} (${m.incomingCount} incoming)`);
    }
```

**Risk:** Wenn der Top-Hub-Block aktiviert wird (FEAT-03-26), landen Note-Summaries der Top-30 Hubs im stabilen System-Prompt-Prefix bei jeder LLM-Conversation. Das LLM-Provider (Anthropic, OpenAI etc) sieht den Block. Wenn ein User personenbezogene Notes (Tagebuch, Patient-Notes, vertrauliche Geschaeftsinformationen) als Hub-Notes hat, werden deren Summaries an den LLM-Provider geleakt. Setting-gated mit Default off, aber Settings-UI gibt **keinen Hinweis** auf den Privacy-Trade-Off.

**Remediation:**
1. Settings-UI-Hinweis bei Top-Hub-Block-Toggle: "Note-Summaries der Top-30 Hubs werden bei jeder Conversation an den LLM-Provider gesendet. Pruefe welche Hub-Notes vertrauliche Daten enthalten."
2. Optional: Cluster-Allowlist (User markiert welche Cluster im Hub-Block landen duerfen).

**Status:** Deferred (Top-Hub-Block ist noch nicht im ContextComposer gewired, kein Live-Risiko bis zur naechsten Wiring-Iteration).

### L-1: source_uri persistiert mit Query-Params (Deferred)

**Severity:** Low
**CWE:** CWE-532 Insertion of Sensitive Information into Log File
**Location:** `src/core/ingest/IngestTriageLogStore.ts:30, 49`

**Risk:** Wenn der Agent eine URL mit sensiblen Query-Params triagt (zB OAuth-Token-Redirect-URL, Session-ID), wird der full source_uri in `ingest_triage_log` persistiert. DB-Inhalt ist beim User-Vault, aber bei Backup-Sharing oder Multi-Device-Sync potentiell exposed.

**Remediation:** Sanitizer fuer URLs vor Persistierung: bekannte sensitive Param-Names (token, api_key, code, state, sessionid) entfernen oder durch Placeholder ersetzen.

**Status:** Deferred (Backlog).

### L-2: AutoTriggerObserver ohne Rate-Limit (Deferred)

**Severity:** Low
**CWE:** CWE-770 Allocation of Resources Without Limits or Throttling
**Location:** `src/core/ingest/AutoTriggerObserver.ts:35-50`

**Risk:** Bulk-Vault-Operationen (git pull, Vault-Import, Mass-Frontmatter-Edit) triggern den vault.on('create')/('modify')-Listener fuer jede betroffene Note. Wenn Auto-Trigger aktiv und viele Notes die konfigurierte Property tragen, koennten innerhalb Sekunden Hunderte Triage-Calls feuern (LLM-Token-Cost-Explosion). `IngestTriageLogStore.exists()` schuetzt nur gegen Doppel-Trigger derselben Source, nicht gegen Storm.

**Remediation:** Rate-Limit auf Listener-Level: max N Trigger pro Minute, sonst Drop mit Warning. Setting fuer N (Default 10/Min).

**Status:** Deferred (Backlog).

### Info-1: Stufe3PeriodicJob-State nicht persistent (Note)

**Severity:** Info
**Location:** `src/core/health/Stufe3PeriodicJob.ts:67-72`

**Risk:** `Stufe3JobState` wird im Konstruktor angenommen, aber das Plugin laedt/persistiert state aktuell nicht in der DB. Bei Plugin-Reload mid-week wird der Budget-Counter zurueckgesetzt. Theoretisch koennte ein User durch mehrere Plugin-Reloads das Wochen-Budget umgehen.

**Remediation:** state in cluster_metadata-Tabelle (oder neue job_state-Tabelle) persistieren. Wiring beim Plugin-onload.

**Status:** Info (Stufe-3-Job ist noch nicht setInterval-wired, kein Live-Risiko).

## Positive Findings

- **SQL-Injection-Schutz konsequent:** alle BA-25 Stores nutzen ausschliesslich parameterized Queries (`?`-Placeholders). Selbst Bulk-Operations mit dynamischer Anzahl Placeholders bauen die `?`-Strings sicher (`notePaths.map(() => '?').join(',')`).
- **Atomic Frontmatter-Write:** FrontmatterWriter nutzt `app.fileManager.processFrontMatter` (atomic) plus optional WriterLock (ADR-79) im obsidian-sync-Mode. Kein YAML-Korruption-Risiko.
- **Token-Budget hart-cap:** Stufe3PeriodicJob bricht bei Budget-Erschoepfung ab, kein soft cap. Vorhersagbare Maximalkosten.
- **Default-konservativ:** alle BA-25-Toggles default off, kein silent rollout. User-Trust gewahrt.
- **MOC-Marker SHA-Detection:** MOCMaintainer prueft SHA gegen User-Modifikation, schuetzt User-Edits vor unbeabsichtigtem Auto-Overwrite.
- **No eval/Function/innerHTML:** keine dynamische Code-Execution, keine DOM-XSS-Vektoren.
- **No new Dependencies:** SCA-Phase trivial, keine zusaetzlichen Drittanbieter-Pakete.
