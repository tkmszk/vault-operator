### High (P1)

**H-S1: fast-uri (MCP-SDK transitive) -- CWE-22 Path Traversal**

- Severity: High (CVSS 7.5)
- CWE: CWE-22 Path Traversal via percent-encoded dot segments
- Location: `node_modules/fast-uri@3.1.0` (transitive via
  `@modelcontextprotocol/sdk@1.29.0` -> `ajv@8.18.0` -> `fast-uri`)
- Risk: `fast-uri` parsed URIs fuer das MCP-SDK. Eine percent-
  encoded Dot-Segment (`%2e%2e`) kann die normale Path-Traversal-
  Validation umgehen. Im Obsilo-Kontext wird MCP fuer Cross-Surface-
  Tools (ChatGPT-Connector, Cloudflare DO Relay) genutzt; ein
  malicious MCP-Server koennte ueber tool-args mit eingebettetem
  URI Path-Traversal triggern. Konkrete Exploitation haengt davon
  ab ob/wie obsilo URIs aus Tool-Args weiterreicht. Worst-Case:
  Vault-Read ausserhalb des Vault-Roots ueber MCP-Boundary.
- Affected version: `fast-uri@3.1.0`, fix in `>=3.1.1`. Advisory:
  GHSA-q3j6-qgpj-74h6, GHSA-v39h-62p7-jpjc.
- Remediation: Override in `package.json`:
  ```json
  "overrides": {
    "fast-uri": ">=3.1.2"
  }
  ```
  Plus `npm install` und Re-Audit. Falls MCP-SDK selbst eine neuere
  ajv mit-bringt, alternativ `@modelcontextprotocol/sdk` auf
  neueste Version updaten.
- Status: Confirmed. Production-Pfad (MCP).

**H-S2: fast-xml-builder (AWS Bedrock transitive) -- CWE-611 XXE**

- Severity: High
- CWE: CWE-611 Improper Restriction of XML External Entity
  Reference (plus CWE-91 XML Comment Injection, moderate)
- Location: `node_modules/fast-xml-builder@1.1.5` (transitive via
  `@aws-sdk/client-bedrock-runtime` -> `@aws-sdk/core` ->
  `@aws-sdk/xml-builder` -> `fast-xml-parser` -> `fast-xml-builder`)
- Risk: AWS-SDK Bedrock nutzt fast-xml-builder fuer Request-Build.
  Bei Bedrock-Provider-Mode sendet obsilo XML-Requests. Wenn ein
  Angreifer Eingaben kontrolliert, die in einem XML-Attribut
  landen, kann der CVE Attribute-Werte mit unerwarteten Quotes
  einschleusen und so XXE oder Comment-Injection ausloesen.
  In der Praxis: Provider-Selection ist User-Setting, der Bedrock-
  Provider wird nur aktiviert wenn explizit gewaehlt. Daher
  Eintrittswahrscheinlichkeit niedriger, Exploitability aber hoch
  bei aktivem Bedrock.
- Affected: `fast-xml-builder@1.1.5`, fix in `>=1.1.7`. Advisories:
  GHSA-5wm8-gmm8-39j9, GHSA-45c6-75p6-83cc.
- Remediation: Override in `package.json`:
  ```json
  "overrides": {
    "fast-xml-builder": ">=1.1.7"
  }
  ```
- Status: Confirmed. Production-Pfad nur bei AWS-Bedrock-Provider.

### Medium (P2)

**M-S1: hono (MCP-SDK transitive) -- 5 CVEs unter Override-Lock**

- Severity: Moderate (kombiniert)
- CWE: CWE-400 (DoS bodyLimit bypass), CWE-74 (HTML/CSS Injection
  in JSX), CWE-1284 (JWT NumericDate validation), CWE-524
  (Cache-Middleware leaks across users)
- Location: `node_modules/hono@4.12.14` (transitive via
  `@modelcontextprotocol/sdk` -> `@hono/node-server` -> `hono`)
- Risk: Aktueller Override `"hono": ">=4.12.14"` erlaubt
  vulnerable Versionen. Die Vulns betreffen JSX-SSR (obsilo
  nutzt das nicht direkt), Cache-Middleware (nicht direkt),

  pro FEATURE-1403, kein JWT verify in der Hot-Path), bodyLimit
  bypass (nur relayWorker auf Cloudflare DO, dort ist 1 MB Body-
  Cap im DO selbst gesetzt). Praktischer Impact niedrig, aber
  best-practice ist Update.
- Remediation: Override-Bump in `package.json`:
  ```json
  "overrides": {
    "hono": ">=4.12.18",
    "@hono/node-server": ">=1.19.10"
  }
  ```
- Status: Confirmed. Production-Pfad (MCP-Relay).

**I-2: PKCE Loopback bindet auf 127.0.0.1**

- Location: `src/core/auth/PkceLoopbackServer.ts:85`
- Beobachtung: Der HTTP-Server fuer den OAuth-Callback bindet
  explizit auf `127.0.0.1`, nie auf `0.0.0.0`. Single-shot
  Callback, danach destroyed. Kein Listener-Drift, kein Local-Net-
  Exposure.
- Status: Positive Finding -- secure-by-default umgesetzt.

**I-3: ChatGptOAuth nutzt SafeStorage und crypto.randomBytes**

- Location: `src/core/auth/ChatGptOAuthService.ts:418,425`
- Beobachtung: PKCE-State und -Verifier werden ueber
  `crypto.randomBytes(numBytes).toString('base64url')` erzeugt
  (CSPRNG, nicht Math.random). Tokens werden ueber
  `SafeStorageService.encrypt()` persistiert (Electron-platform-
  encryption). Code-Hash ueber SHA-256 mit `base64url`-Encoding.
- Status: Positive Finding -- konformer PKCE-Flow.

**I-4: FIX-19-28-06 Regex ohne ReDoS-Anfaelligkeit**

- Location: `src/core/tools/vault/IngestDocumentTool.ts`
- Pattern A: `\[\[[^\]]+\|↗\]\](?:\s+\^[A-Za-z0-9_-]+)?\s*$`
- Pattern B: `\[\[([^#\]|]+)#Page\s+(\d+)\|↗\]\]`
- Beobachtung: Beide Patterns nutzen negierte Character Classes
  (`[^\]]+`, `[^#\]|]+`) mit klarer Termination. Optionale Gruppe
  `(?:\s+\^[A-Za-z0-9_-]+)?` ist nicht-nested. Kein Catastrophic
  Backtracking moeglich.
- Status: Positive Finding.
