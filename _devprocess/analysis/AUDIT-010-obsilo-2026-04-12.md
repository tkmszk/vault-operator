## Findings (nach Prioritaet)

### P2: Should Fix (Medium)

#### M-1: Path Traversal in AssetProvisioner (CWE-22)

| Field | Value |
|-------|-------|
| **Severity** | Medium |
| **CWE** | CWE-22 |
| **Location** | `src/core/AssetProvisioner.ts:40-41` |
| **Status** | Resolved -- `..` und `/` Check hinzugefuegt |

**Finding:** `relativePath` aus dem embedded-assets JSON wird ohne Validierung
in den Dateipfad konkateniert: `` `${pluginDir}/${relativePath}` ``. Ein Eintrag
wie `../../../.bashrc` koennte aus dem Plugin-Verzeichnis ausbrechen.

**Risk:** Gering in der Praxis -- die Daten kommen aus `embedded-assets.json`,
das zur Build-Time aus kontrollierten Verzeichnissen generiert wird. Ein Angriff
erfordert Kompromittierung des Build-Systems. Defense-in-Depth erfordert trotzdem
Validierung.

**Remediation:**

```diff
  for (const [relativePath, content] of Object.entries(assets)) {
+     // Defense-in-depth: reject path traversal attempts
+     if (relativePath.includes('..') || relativePath.startsWith('/')) {
+         console.warn(`[AssetProvisioner] Rejected unsafe path: ${relativePath}`);
+         continue;
+     }
      const fullPath = `${pluginDir}/${relativePath}`;
```

#### M-2: CDN WASM Download ohne Integritaetspruefung (CWE-829)

| Field | Value |
|-------|-------|
| **Severity** | Medium |
| **CWE** | CWE-829 |
| **Location** | `src/core/knowledge/KnowledgeDB.ts:204-210` |
| **Status** | Resolved -- SHA-256 Integrity Check nach CDN Download |

**Finding:** sql-wasm.wasm wird von `cdn.jsdelivr.net` heruntergeladen und
direkt auf Disk gecacht ohne SHA-256 Hash-Pruefung. Im Gegensatz dazu hat
EsbuildWasmManager eine vollstaendige Hash-Verifikation.

**Risk:** CDN-Kompromittierung oder MITM koennte ein manipuliertes WASM-Binary
einschleusen. sql.js fuehrt beliebigen nativen Code aus (SQLite-Engine). Allerdings:
HTTPS schuetzt den Transport, jsdelivr hat eigene Integrity-Checks, und der Download
passiert nur einmal (dann wird gecacht).

**Remediation:**

```diff
+ private static readonly SQL_WASM_SHA256 = '...'; // Hash von sql.js@1.14.1
+
  private async loadWasmBinary(pluginMainDir: string): Promise<Buffer> {
      // ... disk candidates ...
      const response = await requestUrl({ url: cdnUrl });
      const buffer = Buffer.from(response.arrayBuffer);
+
+     // Verify integrity (defense-in-depth against CDN compromise)
+     const hashHex = Array.from(
+         new Uint8Array(await crypto.subtle.digest('SHA-256', response.arrayBuffer))
+     ).map(b => b.toString(16).padStart(2, '0')).join('');
+     if (hashHex !== KnowledgeDB.SQL_WASM_SHA256) {
+         throw new Error('[KnowledgeDB] WASM integrity check failed');
+     }
```

#### M-3: HTTP Response Status nicht geprueft bei CDN Download (CWE-252)

| Field | Value |
|-------|-------|
| **Severity** | Medium |
| **CWE** | CWE-252 |
| **Location** | `src/core/knowledge/KnowledgeDB.ts:205` |
| **Status** | Resolved -- implizit durch M-2 SHA-256 Check geloest |

**Finding:** `requestUrl()` wirft bei non-2xx Status keinen Fehler in allen
Obsidian-Versionen konsistent. Das heruntergeladene `response.arrayBuffer`
koennte eine HTML-Fehlerseite statt WASM-Binary enthalten, wird aber trotzdem
auf Disk gecacht.

**Risk:** Korruptes Cache-File das bei jedem Start sql.js crashen laesst.
User muss manuell loeschen.

**Remediation:** Zusammen mit M-2 fixen -- Hash-Pruefung erkennt automatisch
korrupte Downloads.

### P3: Consider (Low)

#### L-1: Build-Time Asset-Scan ohne Extension-Filter (CWE-426)

| Field | Value |
|-------|-------|
| **Severity** | Low |
| **CWE** | CWE-426 |
| **Location** | `esbuild.config.mjs:36-48` |
| **Status** | Resolved -- Dotfile-Filter hinzugefuegt |

**Finding:** `generateEmbeddedAssets()` liest alle Dateien in `bundled-skills/`
ohne Extension-Whitelist. `.DS_Store`, versehentliche `.env`-Dateien, oder
andere unbeabsichtigte Dateien koennten eingebettet werden.

**Remediation:**

```diff
  for (const file of readdirSync(srcDir)) {
      const filePath = join(srcDir, file);
      if (statSync(filePath).isDirectory()) continue;
+     if (file.startsWith('.')) continue; // Skip dotfiles (.DS_Store etc.)
      assets[`skills/${skillDir}/${file}`] = readFileSync(filePath, "utf-8");
```

#### L-2: Version-Marker ohne Integritaet (CWE-345)

| Field | Value |
|-------|-------|
| **Severity** | Low |
| **CWE** | CWE-345 |
| **Location** | `src/core/AssetProvisioner.ts:29-31` |
| **Status** | Confirmed |

**Finding:** Der `.obsilo-assets-version` Marker enthaelt nur die Versionsnummer.
Ein Angreifer mit Schreibzugriff auf das Plugin-Verzeichnis koennte den Marker
auf die aktuelle Version setzen und damit die Extraktion unterdruecken, waehrend
die Assets manipuliert bleiben.

**Risk:** Sehr gering -- erfordert lokalen Dateisystem-Zugriff. Kein praktisches
Angriffsszenario, da ein Angreifer mit diesem Zugriffslevel direkt main.js
manipulieren koennte.

**Remediation:** Kein Fix noetig. Dokumentation als Known Limitation.

## Positive Findings

- **Build-Time Embedding statt Runtime-Download:** Workers und Skills werden zur
  Build-Time eingebettet, nicht zur Laufzeit aus dem Internet geladen. Nur sql-wasm.wasm
  hat einen CDN-Fallback (und auch der wird lokal gecacht).
- **Version-Gating:** AssetProvisioner extrahiert nur bei Versionsaenderung --
  verhindert unnoetige Disk-Writes und Race Conditions.
- **Graceful Degradation (FIX-18):** KnowledgeDB/MemoryDB werden bei Fehler null-gesetzt,
  statt kaskadierende Exceptions. Defensive Programmierung.
- **require() statt import() (FIX-17):** Korrekte Electron-kompatible Loesung.
- **Bestehende Massnahmen intakt:** Alle Findings aus AUDIT-003 bis AUDIT-009 weiterhin
  korrekt adressiert (Sandbox SHA-256, Path-Traversal Guards, SSRF-Schutz, etc.).
