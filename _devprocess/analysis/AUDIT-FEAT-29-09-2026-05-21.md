## Findings

### L-1: Path-Traversal in snapshotId parameter -- RESOLVED

**Severity:** Low (Defense-in-Depth)
**CWE:** CWE-22 Path Traversal
**Location:** `src/core/skills/SkillSnapshotService.ts:snapshotFolder`
**Status:** Resolved im Audit-Pass

**Risk:** `restore(skillName, snapshotId)`, `tag(skillName, snapshotId, tagName)`, and `untag(skillName, snapshotId, tagName)` accept a caller-supplied `snapshotId`. The helper `snapshotFolder(skillName, id)` concatenates the id into the path without validating its shape. A crafted id like `../../etc/passwd` would resolve outside the skill folder. The `skillName` IS validated via `isSafePathSegment` -- the `snapshotId` was the gap.

Trust boundary: the SkillVersionsModal feeds ids from `list()` which the service emits itself, so the UI path is safe. But the modal is not the only consumer -- agent tools or a future API could pass arbitrary ids. Defense-in-Depth lohnt sich.

**Remediation (applied):** In-helper validation against `[A-Za-z0-9._-]+`:

```typescript
private snapshotFolder(skillName: string, id: string): string {
    if (!/^[A-Za-z0-9._-]+$/.test(id)) {
        throw new Error(`Unsafe snapshot id rejected: ${JSON.stringify(id)}`);
    }
    return `${this.versionsFolder(skillName)}/${id}`;
}
```

**Tests (RED-First):** 2 new tests in `SkillSnapshotService.test.ts > edge cases`:
- restore rejects `../escape` and `/etc/passwd`
- tag / untag reject `../escape`

### I-1: SkillWriteInterceptor recursion guard -- BY DESIGN

**Severity:** Info
**Location:** `src/core/skills/SkillWriteInterceptor.extractSkillName`
**Status:** by design

**Risk:** The interceptor monkey-patches adapter.write. When SkillSnapshotService.snapshot() runs INSIDE the monkey-patched write, it calls adapter.write again to copy files into `.versions/`. Without a recursion guard this would loop.

**Analysis:** `extractSkillName` returns null for any path matching `<skillsRoot>/{name}/.versions/...`. The monkey-patch checks this before calling snapshot. Loop is broken.

Pinned by `SkillWriteInterceptor.test.ts > does NOT snapshot for writes to the .versions/ subfolder itself`. Defensive-by-design, no action needed.

### I-2: Concurrent snapshots produce non-atomic views -- BY DESIGN

**Severity:** Info
**Location:** `src/core/skills/SkillSnapshotService.snapshot`
**Status:** by design

**Risk:** If two adapter.write calls fire concurrently and both pass the 5-second debounce, two snapshot() invocations run in parallel. The walks could see half-old half-new state -- the snapshot becomes a non-atomic view.

**Justification:** Skill writes are normally sequential (single agent loop, single user-edit at a time). The race is theoretical, and the impact is "snapshot is slightly inconsistent" -- not data loss or security issue. ID counter guarantees unique folders so no file collision.

For a future iteration, we could add per-skill mutex. For v1, by-design.

### I-3: SkillVersionsModal uses window.prompt for tag input -- INFO

**Severity:** Info
**Location:** `src/ui/modals/SkillVersionsModal.handleTag`
**Status:** by-design (Phase 1)

**Risk:** Tag input via window.prompt is functional but spartan. A future native Obsidian-style Setting input would be nicer. Not a security issue.

The tag value gets stored in snapshot.json. We don't render it as HTML -- the modal uses `setText` (Obsidian DOM API) which escapes content. No XSS surface.

## Positive Findings

- **isSafePathSegment for skillName** -- enforced at entry of every method via `skillFolder()`. Reserves the same regex as plugin-id validation elsewhere.
- **Snapshot ids emitted by service** -- ISO-timestamp + counter, monotonic and unique per process.
- **Write-error swallowing in interceptor** -- "data over history": if snapshot fails, the user's write still goes through. Logged as warn so dev catches it.
- **Tags survive prune** -- tagged versions are explicitly preserved regardless of retention count. Pinned by tests.
- **Pre-restore snapshot before every restore** -- restore is reversible by design. Pinned by tests.
