# AUDIT-020: FEAT-24-06 MCP-Listing-Cap + read_mcp_tool + Built-in deferred-Review (ADR-118)

## Executive Summary

Geprueft wurden die Code-Aenderungen von FEAT-24-06 / ADR-118 -- MCP-Tool-
Description-Cap in der stabilen System-Prompt-Listung, neues NICHT-deferred
Tool `read_mcp_tool(server, name)` mit voller Description und kompaktem
InputSchema-Summary, plus ein zweiter `deferred`-Pass in der zentralen
Tool-Metadata. Betroffene Quelldateien:
`src/core/prompts/sections/tools.ts`,
`src/core/tools/mcp/ReadMcpToolTool.ts` (neu),
`src/core/tools/ToolRegistry.ts`,
`src/core/tools/types.ts`,
`src/core/tools/toolMetadata.ts`.

Ergebnis: **0 Critical, 0 High, 0 Medium**. 4 Info-Hinweise -- davon einer
ein systemisches Drift-Pattern, das ueber FEAT-24-06 hinaus betrifft und
als IMP-24-06-01 ins BACKLOG geht.

Hauptaudit-Vektoren aus dem /coding-/testing-Handoff sind alle geprueft;
der Path-Traversal-Vektor existiert nicht, die Resource-Consumption-Frage
ist durch die `HARD_TOOL_OUTPUT_CAP_CHARS = 60_000`-Bodenplatte aus
FEAT-24-03 / PLAN-18 bereits abgedeckt, die Suffix-Quote-Robustheit ist
kosmetisch.

Release-Empfehlung fuer die FEAT-24-06-Aenderungen: **Green**. SC-6 bleibt
fuer den manuellen Live-Messlauf reserviert; das ist eine Funktions-, keine
Sicherheitsfrage.

## Methodik

- **SAST:** gezielte Durchsicht der Diffs gegen Path Traversal (CWE-22),
  uncontrolled Resource Consumption (CWE-400), Injection (CWE-94),
  Eingabevalidierung an Vertrauensgrenzen (CWE-20), Insufficient
  Documentation (CWE-1059).
- **OWASP LLM Top 10:** LLM01 (Prompt Injection via Server-controlled
  Tool-Result), LLM02 (Insecure Output Handling), LLM07 (Insecure Plugin
  Design).
- **SCA:** Keine neuen Runtime-Dependencies durch FEAT-24-06. Baseline
  unveraendert gegenueber AUDIT-019 (1 vorbestehender Moderate `mermaid`,
  DEBT-SCA-2026-05-12).
- **Zero Trust:** Vertrauensgrenze zwischen Modell-Input (Tool-Call-
  Argumente), User-Settings (`activeMcpServers`), und MCP-Server-Daten
  (Tool-Liste, Descriptions, Input-Schemas).
- **Drift-Audit:** ToolName-Union vs `TOOL_METADATA`-Schluessel statisch
  abgeglichen, weil FEAT-24-06 das Pattern bei `inspect_self` /
  `update_settings` aufgedeckt hat.

## Findings

### F-1 (Info) -- TOOL_METADATA-Drift: 16 weitere Tools im selben Pattern

**Severity:** Info. **CWE:** CWE-1059 (Insufficient Documentation).
**Status:** Confirmed, deferred zu IMP-24-06-01.

**Location:** `src/core/tools/toolMetadata.ts` (fehlende Schluessel),
`src/core/tools/types.ts` (ToolName-Union).

**Evidence:** statischer Drift-Check ergab 16 Tools in der `ToolName`-Union
ohne `TOOL_METADATA`-Eintrag plus 1 Tool im `TOOL_METADATA` ohne Eintrag
in der Union (`check_presentation_quality`). Die Liste:

```
Fehlende TOOL_METADATA-Eintraege (in Union, nicht im Metadata):
- _memory_atomize, _memory_single_call, anti_echo_search,
  configure_model, create_canvas, ingest_deep, ingest_triage,
  list_memory_source_notes, mark_for_memory,
  mark_note_as_memory_source, read_agent_logs, recall_memory,
  search_history, switch_mode, unmark_note_as_memory_source,
  update_soul

Fehlender Union-Eintrag (in Metadata, nicht in Union):
- check_presentation_quality
```

**Risk:** Diese Tools sind in der `ToolRegistry` registriert; ihre Schemas
(`getDefinition()`) gehen ins `tools`-Feld jeder API-Anfrage; sie sind aber
nicht in der zentralen System-Prompt-Tools-Listung dokumentiert. Folgen:

- Das Modell sieht das JSON-Schema im API-Call, kennt aber nicht die
  fachliche Description/WhenToUse aus der zentralen Tool-Metadata. Das ist
  ein Effizienz-/Nutzbarkeits-Issue, kein direkter Sicherheitspfad.
- `find_tool` koennte sie nicht ranken (Filterung in
  `FindToolTool.execute` ueber `DEFERRED_TOOL_NAMES`-Intersection mit
  `TOOL_METADATA`); aktuell ist keines dieser 16 Tools deferred, weshalb
  `find_tool` heute eh nicht greift -- aber jeder zukuenftige
  `deferred`-Pass auf eines dieser Tools wuerde es unentdeckbar machen
  (genau das Pattern, das FEAT-24-06 bei `inspect_self` /
  `update_settings` aufgedeckt hat).
- `check_presentation_quality` im `TOOL_METADATA` aber nicht in der Union
  ist die spiegelbildliche Drift: Type-Check waere strikter, wenn beide
  Listen automatisch synchron gehalten wuerden.

**Remediation:** Eigener V-Model-Pass (kein Sicherheitsthema, sondern
Code-Hygiene): Audit jedes der 16 Tools pro Eintrag und entscheiden, ob
es eine `TOOL_METADATA`-Beschreibung braucht (alle aktiv genutzten Tools
sollten eine haben). `check_presentation_quality`-Spiegelfall ebenfalls
in dem Pass mit-pruefen. Als IMP-24-06-01 (`feature: FEAT-24-06`,
`epic: EPIC-24`, `audit-refs: AUDIT-020`, Source SEC, P3) im BACKLOG.

**Kein direkter Sicherheitsimpact heute.** Die Tools sind ueber die
Registry funktional erreichbar; nur die zentrale Tools-Section im
System-Prompt zeigt sie nicht namentlich an.

### F-2 (Info) -- Schema-Summary-Renderer ohne eigenen Per-Property-Cap

**Severity:** Info. **CWE:** CWE-400 (Uncontrolled Resource Consumption).
**Status:** Confirmed, accepted (Defense-in-Depth bereits via Pipeline-Cap).

**Location:** `src/core/tools/mcp/ReadMcpToolTool.ts`
(`renderInputSchemaSummary` + `describeType`).

**Risk:** Ein boesartiger oder buggy MCP-Server koennte ein
`inputSchema` mit sehr vielen Properties (tausende), sehr langen
Property-Namen, oder tief verschachtelten Array-Typen liefern. Der
Renderer iteriert die Properties ohne lokalen Cap und produziert eine
Zeile pro Eintrag.

**Mitigation bereits vorhanden:**
`ToolExecutionPipeline.capOversizedToolOutput()` mit
`HARD_TOOL_OUTPUT_CAP_CHARS = 60_000` (FEAT-24-03 / PLAN-18) kappt am
Pipeline-Ausgang. Auch ein extrem grosses Schema-Output landet nie
ungekappt in der History. Damit ist die Vertrauensgrenze gegen
boesartige Server eingehalten.

**Remediation (optional, kein Muss):** Ein lokaler Cap im Renderer
(z.B. max 50 Properties, danach `... (truncated, N more properties)`)
wuerde die Lesbarkeit am LLM-Ende erhoehen und vermeiden, dass die
Pipeline-Bodenplatte mitten in einer Property abschneidet. Reine
Defense-in-Depth, kein direkter Sicherheitsgewinn.

**Status:** Accepted, kein BACKLOG-Eintrag noetig. Falls der Live-
Messlauf von SC-6 zeigt, dass MCP-Server in der Praxis solche
Schema-Bomben liefern, lohnt sich der Renderer-Cap als IMP. Heute
nicht.

### F-3 (Info) -- Whitelist-Check 1:1 zwischen `UseMcpToolTool` und `ReadMcpToolTool` dupliziert

**Severity:** Info. **CWE:** CWE-1004 (Sensitive Cookie without HttpOnly --
hier in der Code-Smell-Form, Duplizierung von Sicherheitslogik).
**Status:** Confirmed, accepted.

**Location:** `src/core/tools/mcp/UseMcpToolTool.ts:72-81` und
`src/core/tools/mcp/ReadMcpToolTool.ts` (analoge Block-Struktur).

**Risk:** Wenn der Whitelist-Check in einer der beiden Stellen geandert
wird, divergieren `use_mcp_tool` und `read_mcp_tool` in ihrer
Sicherheitslogik. Heute identisch. Kein Eskalations-Pfad: beide checken
denselben `activeMcpServers`-Satz.

**Remediation (optional):** Extraktion in einen gemeinsamen Helper
`assertMcpServerAllowed(plugin, server)` in `src/core/mcp/` oder als
private statische Methode auf einem `McpToolBase`. Sehr klein, hat aber
mittelfristig Sinn. Kein BACKLOG-Eintrag heute; falls ein dritter
MCP-Tool entsteht, ist die Extraktion der natuerliche Zeitpunkt.

**Status:** Accepted, no action required.

### F-4 (Info) -- MCP-Description-Cap-Suffix mit Quotes nicht-escapend

**Severity:** Info. **CWE:** keine (kosmetische Robustheit).
**Status:** Confirmed, accepted (theoretischer Fall).

**Location:** `src/core/prompts/sections/tools.ts`, `capMcpDescription`.

**Risk:** Der Suffix-String `... [full description: read_mcp_tool({
server: "X", name: "Y" })]` setzt Server- und Tool-Namen ohne Escape in
doppelte Anfuehrungszeichen ein. Enthielte ein Server-Name oder ein
Tool-Name ein `"`-Zeichen, waere die Zeile syntaktisch leicht kaputt.
Auswirkung: das Modell wuerde den `read_mcp_tool`-Hint mit kaputter
JSON-Syntax sehen, koennte aber den Sinn aus dem umgebenden Text
weiterhin erfassen.

**Realitaetscheck:** Server-Namen kommen aus User-Settings
(Obsidian-MCP-Konfiguration), Tool-Namen aus dem MCP-Server selbst. In
der Praxis sind beide kebab-case oder snake_case ohne Anfuehrungszeichen.
Quotes als Teil eines Namens sind nie beobachtet worden; semantisch
abwegig.

**Remediation (optional):** `JSON.stringify(server)` + `JSON.stringify(name)`
statt `"${server}"` + `"${name}"`. Eine Zeile Aenderung, kein Aufwand.
Reine Robustheit, kein Sicherheitsproblem.

**Status:** Accepted, no action required.

## SCA

Keine neuen Runtime-Dependencies durch FEAT-24-06. `npm audit`-Baseline
unveraendert gegenueber AUDIT-019: 1 vorbestehender Moderate-Eintrag
(`mermaid` Gantt-/classDef-Advisories, transitiv, DEBT-SCA-2026-05-12),
NICHT durch FEAT-24-06 verursacht.

## Positivbefunde

- **Klare Vertrauensgrenze in `ReadMcpToolTool`:** Modell-Input `server`
  und `name` sind reine Lookup-Keys, kein Filesystem-Pfad-Vektor. Analog
  zum read_skill-Pattern aus FEAT-24-09.
- **Whitelist-Enforcement fail-closed:** ohne `activeMcpServers`-Eintrag
  weist `read_mcp_tool` die Anfrage zurueck (`No active MCP server set`
  -> Fehler). Kein "default deny missing" oder Skip.
- **Connection-Status-Check fail-closed:** Server muss tatsaechlich
  verbunden sein; sonst Fehler mit Liste der verbundenen Server.
- **Defense-in-Depth durch Pipeline-Cap:** `HARD_TOOL_OUTPUT_CAP_CHARS = 60_000`
  in `ToolExecutionPipeline.capOversizedToolOutput()` aus FEAT-24-03 /
  PLAN-18 ist die zweite Verteidigungslinie gegen jede Form von
  Tool-Result-Bloat (auch wenn der Renderer selbst keinen lokalen Cap
  hat).
- **Konsistenz zum Skill-Pattern (ADR-116):** `read_mcp_tool` folgt
  exakt dem Verzeichnis-im-stabilen-Block + Detail-on-demand-Muster.
  Eine konsistente Architektur reduziert die kognitive Surface fuer
  zukuenftige Tools im selben Muster.
- **MCP-Description-Cap deterministisch:** der `capMcpDescription`-Helper
  cuttet immer an Position 200 (mit `trimEnd()` fuer trailing whitespace)
  und produziert deterministische Strings -- der gecachte
  System-Prompt-Praefix bleibt cache-stable.
- **Reduzierte `tools`-Feld-Surface:** zwei zusaetzliche Built-ins
  (`inspect_self`, `update_settings`) sind jetzt deferred. Ein extrem
  selten benoetigtes Schema verschwindet aus jedem API-Call und wird
  nur bei Bedarf via `find_tool` aktiviert.
- **Tests-Verankerung:** SC-3 (`read_mcp_tool` NOT deferred + group=mcp)
  und SC-4 (Built-in-deferred-Pass) sind als Regression-Assertions in
  `deferredToolLoading.test.ts` verankert. Drift-Schutz.

## Architektonische Folgepunkte

Keiner kritisch fuer FEAT-24-06. F-1 (TOOL_METADATA-Drift) ist ein
querschnittliches Hygiene-Item, das aus diesem Audit hervorgeht und
als IMP-24-06-01 ein eigenes V-Model-Item bekommt. Mittel- bis
langfristig waere eine automatische ToolName-Union / TOOL_METADATA
Konsistenz-Pruefung (TypeScript-Type oder Vitest-Assertion) sinnvoll,
damit das Drift-Pattern nicht wiederkehrt. Im IMP-24-06-01 mit-erwaegen.
