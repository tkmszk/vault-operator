## Findings

### L-1: Path-Traversal in BuiltinSkillMaterializer relPath -- RESOLVED

**Severity:** Low (Defense-in-Depth)
**CWE:** CWE-22 Path Traversal
**Location:** `src/core/skills/BuiltinSkillMaterializer.ts:80-110`
**Status:** Resolved im Audit-Pass

**Risk:** `materializeAll` iteriert ueber `Object.entries(bundle)` mit jedem Eintrag als `[skillName, files]` und dann `Object.entries(files)` mit `[relPath, content]`. Der `skillName` wurde via `isSafePathSegment` validiert, der `relPath` aber nicht. Ein boeser `relPath` wie `"../../escape.md"` ergibt nach `normalizePath(`${targetDir}/${relPath}`)` einen Pfad ausserhalb des Skill-Ordners.

Trust-Boundary: `BUNDLED_SKILLS` wird zur Build-Zeit aus `bundled-skills/` via esbuild generiert. Der Inhalt ist Sebastian-managed im Repo, keine User-Eingabe -- praktisches Risiko = niedrig. Defense-in-Depth lohnt sich trotzdem.

**Remediation (applied):** Pre-Validierung des relPath in `materializeAll`:
- `relPath.includes('..')` -> reject
- `relPath.startsWith('/')` oder `'\\'` -> reject
- `relPath.includes('\0')` -> reject (NUL-byte path-injection)
- Post-normalize Containment-Check: `fullPath.startsWith(targetDir + '/')` mit Fail-Soft (Error in report, skill landing weiter, andere Files aus dem Skill skipped).

```typescript
if (relPath.includes('..') || relPath.startsWith('/') || relPath.startsWith('\\') || relPath.includes('\0')) {
    report.errors.push({ name: skillName, reason: `unsafe relpath rejected: ${relPath}` });
    continue;
}
const fullPath = normalizePath(`${targetDir}/${relPath}`);
if (!fullPath.startsWith(`${targetDir}/`) && fullPath !== targetDir) {
    report.errors.push({ name: skillName, reason: `path escapes skill folder: ${relPath}` });
    continue;
}
```

**Tests (RED-First):** 4 neue Tests in `BuiltinSkillMaterializer.test.ts > AUDIT-FEAT-29-11 L-1 path-traversal guard`:
- rejects `../escape.md`
- rejects absolute `/etc/passwd`
- rejects NUL-byte `foo\0.js`
- accepts well-formed `scripts/build.js` + `references/notes.md`

### L-2: Potential Zip-Slip in SkillsTab.addFolderToZip -- RESOLVED

**Severity:** Low (Defense-in-Depth)
**CWE:** CWE-22 Path Traversal (Zip-Slip variant)
**Location:** `src/ui/settings/SkillsTab.ts:476-510`
**Status:** Resolved im Audit-Pass

**Risk:** Der Export packt rekursiv den Skill-Folder via `adapter.list(dir)`. Wenn Obsidian's Vault-Adapter jemals einen Pfad mit `..` zurueckgibt (Symlink? Future-Adapter?), wuerde JSZip diesen 1:1 als zip-entry path uebernehmen. Beim Entpacken auf User-Seite koennte das ein Zip-Slip werden ("CVE-2018-1000049 / `evil.zip` extraktion ueberschreibt /etc/passwd"). Obsidian's aktuelle Adapter machen das nicht, aber Containment-Check ist guenstig.

**Remediation (applied):** Containment-Pre-Check pro File und pro Subfolder vor dem `zip.file`-Aufruf:

```typescript
const dirPrefix = `${dir}/`;
if (!filePath.startsWith(dirPrefix)) { console.warn(...); continue; }
const name = filePath.slice(dir.length + 1);
if (name.includes('..') || name.startsWith('/')) { console.warn(...); continue; }
```

Fail-Soft mit `console.warn` und `continue` statt throw -- der Rest des Exports laeuft weiter. Logging ist defensiv: wenn das jemals greift, weiss man sofort dass ein Adapter aus dem Pfad-Konvention ausschert.

### I-1: Base64-Decode-DoS bei extrem grossen Bundle-Assets -- BY DESIGN

**Severity:** Info
**Location:** `BuiltinSkillMaterializer.decodeBase64`
**Status:** By Design

**Risk:** Ein extrem grosses bundle-asset (z.B. 1 GB PNG mit `__b64__` Suffix) wuerde beim Decode `Buffer.from(b64, 'base64')` plus `Uint8Array.charCodeAt`-Loop ausfuehren. Memory-Spike + UI-Lag im Plugin-Start.

**Justification:** `BUNDLED_SKILLS` wird zur Build-Zeit von Sebastian's Repo generiert. Praktisches Risiko = null. Wenn ein Asset zu gross wird, faellt es zuerst im `npm run build`-Pass auf (main.js wird > 5 MB-Ceiling). Keine Aktion noetig.

### I-2: Esbuild liest filesystem-Files in bundled-skills/ -- BY DESIGN

**Severity:** Info
**Location:** `esbuild.config.mjs:walkSkill`
**Status:** By Design

**Risk:** Der recursive scan ruft `readFileSync(filePath)` ohne Symlink-Check. Ein Symlink in `bundled-skills/` koennte auf eine beliebige File ausserhalb des Repos zeigen und deren Inhalt in `main.js` exfiltrieren.

**Justification:** Build-time Trust-Boundary. Die `bundled-skills/`-Files sind im Repo gecheckt und Code-Review-pflichtig. Symlinks waeren PR-Review-Red-Flag. Keine Aktion noetig.
