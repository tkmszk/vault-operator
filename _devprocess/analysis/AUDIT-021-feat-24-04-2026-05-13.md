# AUDIT-021: FEAT-24-04 Subagent-Delegation (profile=research + Per-Call-Token-Budget)

## Executive Summary

Geprueft wurden die Code-Aenderungen von FEAT-24-04 / ADR-113 -- neues
Subagent-Profile-Konzept mit einem `research`-Profile (read-only,
schlanke roleDefinition, reduzierte Tool-Allowlist), neuer optionaler
`profile`-Parameter im `new_task`-Tool, harter Per-Call-Token-Budget
(Default 8000 via `subtaskTokenBudget`). Betroffene Quelldateien:
`src/core/agent/subagent-profiles.ts` (neu), `src/core/tools/agent/NewTaskTool.ts`,
`src/core/tools/agent/newTaskValidation.ts`, `src/core/AgentTask.ts`,
`src/core/systemPrompt.ts`, `src/core/prompts/sections/modeDefinition.ts`,
`src/core/prompts/sections/tools.ts`,
`src/core/prompts/sections/toolDecisionGuidelines.ts`,
`src/core/tools/types.ts`, `src/core/tools/toolMetadata.ts`,
`src/types/settings.ts`.

Ergebnis: **0 Critical, 0 High, 0 Medium**. 3 Info-Hinweise als
Pre-Emptive Notes fuer kuenftige Erweiterungen (kein heutiges Risiko,
kein BACKLOG-Eintrag erforderlich).

Hauptaudit-Vektoren aus dem /coding-/testing-Handoff alle clean oder
mehrfach mitigiert. Profile-Spawn ist eine deutlich engere Vertrauens-
grenze als der Tier-4-Pfad: kein MCP, keine Skills, keine Rules, keine
Plugin-Skills, nur eine eingeschraenkte Read-only-Tool-Allowlist.

Release-Empfehlung fuer die FEAT-24-04-Aenderungen: **Green**. SC-6
(Live-Messlauf des Eltern-Kontext-Wachstums) bleibt fuer manuelle
Vault-Session reserviert; das ist eine Funktions-, keine Sicherheits-
frage.

## Methodik

- **SAST:** gezielte Durchsicht der Diffs gegen Path Traversal (CWE-22),
  Uncontrolled Resource Consumption (CWE-400), Injection (CWE-94),
  Eingabevalidierung an Vertrauensgrenzen (CWE-20), Improper Privilege
  Management (CWE-269).
- **OWASP LLM Top 10:** LLM01 (Prompt Injection via Subagent-Inhalt),
  LLM02 (Insecure Output Handling im Tool-Result an den Parent),
  LLM06 (Sensitive Information Disclosure ueber Profile-Definition),
  LLM07 (Insecure Plugin Design fuer den neuen Tool-Parameter).
- **SCA:** keine neuen Runtime-Dependencies durch FEAT-24-04.
  `npm audit`-Baseline unveraendert gegenueber AUDIT-020.
- **Zero Trust:** Vertrauensgrenze zwischen Modell (Tool-Call-Argumente),
  Profile-Konstanten (Code-defined), Settings (`subtaskTokenBudget`,
  `activeMcpServers`).
- **Statischer Konsistenz-Check:** Profile-Allowlist wirklich kein write
  oder MCP enthaelt, durch `subagent-profiles.test.ts` als Regression
  verankert.

## Findings

### F-1 (Info) -- Pre-Emptive: heutige Profile sind Code-konstant; user-eingebbare Profile waeren ein Prompt-Injection-Vektor

**Severity:** Info. **CWE:** CWE-94 (Code Injection, latent).
**Status:** Acknowledged, future-only.

**Location:** `src/core/agent/subagent-profiles.ts` (Profile-Registry).

**Risk:** Das `RESEARCH_PROFILE` ist eine Konstante im Code. Sein
`roleDefinition`-String geht 1:1 in den Subagent-System-Prompt. Heute
gibt es keinen Pfad, ueber den ein User oder das Modell den
roleDefinition-Inhalt aendern koennte; ein Prompt-Injection-Vektor
existiert nicht.

**Pre-Emptive Note fuer Folge-Items:** wenn ein zukuenftiges Item
user-konfigurierbare Profile einfuehrt (z.B. Settings-UI mit
benutzerdefinierten Profil-Definitionen), waere der `roleDefinition`-
String ein direkter Prompt-Injection-Vektor. Dann brauecht es eine
explizite Sanitization-Strategie (oder Vorgehen wie bei `manage_skill`:
User-content bewusst akzeptiert, mit klar dokumentierter
Trust-Boundary).

**Status:** Acknowledged, no action required heute. Dokumentiert hier,
damit ein spaeterer Audit den Vektor nicht uebersieht.

### F-2 (Info) -- Token-Budget-Schaetzung via chars/4 ist konservativ-grob

**Severity:** Info. **CWE:** keine (Schaetz-Genauigkeit).
**Status:** Confirmed, accepted.

**Location:** `src/core/tools/agent/NewTaskTool.ts` (Budget-Check:
`Math.ceil(message.length / 4)`).

**Risk:** Die Budget-Schaetzung nutzt die Rule of Thumb "4 Chars pro
Token", wie in `src/types/model-registry.ts:estimatePromptTokens`. Reale
Tokenizer (Anthropic, OpenAI) liegen typischerweise zwischen 3 und 4.5
Chars pro Token; deutsche Texte sind in der Praxis dichter, englische
Texte leichter. Ein "8000 Token Budget" entspricht hier 32000 Chars; der
echte Token-Count beim Provider kann +/- 20 Prozent abweichen.

**Folge:** der Budget-Check kann falsch-positiv (Anwender bekommt
Fehler trotz unterhalb-des-echten-Limits) oder falsch-negativ (knapp
ueberschritten, aber Schaetzung sagt OK). In der Praxis fuer den Budget-
Use-Case "Defense-in-Depth gegen riesige Subtask-Messages" ausreichend
genau: der typische Missuse-Fall (eine 100k-Char-Message) wird sicher
gefangen; ein 7990-Token-Edgecase wird vielleicht waeruf 8200 echte
Tokens kommen, aber der Spawn passiert dann mit ueberschaubarer
Toleranz.

**Mitigation alternativ:** Provider-spezifischer Tokenizer oder
`countTokens`-Hilfsfunktion aus dem Anthropic-SDK. Nicht-trivial,
provider-spezifisch, und der Budget ist eine Bodenplatte, kein
Cost-Counter. Nicht hier.

**Status:** Accepted, no action required.

### F-3 (Info) -- Profile-Mode-Override-UX: `mode` wird verworfen wenn Profile gesetzt

**Severity:** Info. **CWE:** keine (UX).
**Status:** Confirmed, by design.

**Location:** `src/core/AgentTask.ts` `spawnSubtask`
(`initialMode: profile ? 'agent' : childMode`).

**Risk:** Wenn das Modell `new_task({ mode: 'ask', profile: 'research', message: '...' })`
aufruft, wird `mode` verworfen und der Subagent laeuft als
`mode: 'agent'` mit Profile-Override. Das ist Profile-Semantik (Profile
bestimmt Scope), kann aber fuer ein Modell, das das Schema nicht ganz
gelesen hat, ueberraschend sein. Die Tool-Description in
`NewTaskTool.getDefinition` adressiert das explizit:

```
"mode": "Sub-agent mode: ... With profile=\"research\", mode is ignored
and the profile runs read-only."
```

Damit ist die UX dokumentiert. Kein Sicherheitsrisiko (der
Profile-Scope ist die engere Vertrauensgrenze; "ask"-Spawn waere
ohnehin read-only und ohne profile waere die Tool-Liste hoeher).

**Status:** Confirmed, by design.

### Hauptaudit-Vektoren (alle clean oder mitigiert)

#### Path-Traversal in spawnSubtask

**Status:** Existiert nicht. `getSubagentProfile(profileName)` ist ein
String-Lookup in einer Code-konstanten `Record<string, SubagentProfile>`-
Map. Kein Filesystem-Pfad wird je aus User-/Modell-Input konstruiert.
Profile-Spawn hat keinen Filesystem-Surface.

#### Resource Consumption via Subtask-Message-Bombe

**Status:** Mitigiert auf zwei Ebenen.

1. **Per-Call-Token-Budget** im NewTaskTool: `Math.ceil(message.length / 4) > budget`
   -> Hard-Stop mit `formatError`, kein Spawn. Default 8000 Tokens via
   `settings.advancedApi.subtaskTokenBudget` mit konstanter Fallback.
2. **Pipeline-Bodenplatte** `HARD_TOOL_OUTPUT_CAP_CHARS = 60_000` aus
   FEAT-24-03 / PLAN-18 kappt das Subagent-Antwort-Tool-Result, sollte
   der Subagent eine grosse Antwort produzieren.

Beide Pfade unabhaengig voneinander aktiv.

#### Prompt Injection via Subagent-Inhalte (LLM01)

**Status:** Acceptable, by design.

- **roleDefinition (Profile):** im Code definiert, keine User-Eingabe.
  Kein Injection-Vektor heute (siehe F-1 fuer Pre-Emptive Note).
- **`message` (Modell-Input an Subagent):** geht in die User-Message
  des Subagent-Loops, nicht in dessen System-Prompt. Standard-Trust-
  Boundary fuer User-Messages; nicht neu in FEAT-24-04.
- **Subagent-Antwort (zurueck an Parent):** das verdichtete Result
  kommt via `pushToolResult` als Tool-Result-Block in die Parent-
  History. Vertrauensgrenze identisch zu jedem anderen Tool-Result
  (alle Tool-Results gehen ueber den Pipeline-Cap, alle koennen
  potenziell adversarial sein). FEAT-24-04 macht die Grenze nicht
  duerfft sie aber auch nicht enger als andere Tools.

#### Privilege Escalation via Profile-Allowlist (CWE-269)

**Status:** Existiert nicht. Profile-Allowlist ist eine Untermenge der
registrierten Tools. `rebuildPromptCache` filtert in dieser Reihenfolge:

1. Mode-Tool-Set holen (alle Tools des Modes).
2. **Profile-Allowlist-Filter:** baseTools = baseTools.filter(t => allowSet.has(t.name)).
3. Deferred-Filter: cachedTools = baseTools.filter(t => !isDeferredTool(t.name)).
4. Activated-Deferred-Tools injizieren: `baseTools.find(t => t.name === name)`
   -- wenn das aktivierte Tool nicht in `baseTools` (also nach Profile-
   Filter), wird es NICHT injiziert.

Damit gewinnt die Profile-Allowlist auch gegen `find_tool`-Aktivierungen.
Selbst wenn ein Profile-Subagent `find_tool` aufrufen koennte (was er
nicht kann, weil `find_tool` nicht in der research-Allowlist ist),
koennte ein per find_tool aktiviertes Tool nicht in den Subagent-tools
gelangen. Mehrfach gesichert.

#### MCP-Isolation des Profile-Subagent

**Status:** Doppelt-gesichert.

- **`mcpClient: undefined`** im `childTask.run({...})`-Aufruf wenn
  Profile gesetzt. Damit hat der Subagent keinen MCP-Client.
- **`use_mcp_tool`/`read_mcp_tool` NICHT in research-Allowlist.** Das
  Tool-Schema des Subagent enthaelt diese Tools also gar nicht.

Beide Pfade unabhaengig. Selbst wenn einer ausfaellt (z.B. ein zweites
Profile vergisst, mcpClient zu unterbinden), schliesst die Allowlist
den Zugriff weiter.

#### Subagent-Nesting

**Status:** Mehrfach gesichert.

- `maxSubtaskDepth`-Logik (heute Default 2) unveraendert; Profile-
  Subagents zaehlen normal.
- `new_task` NICHT in research-Profile-Allowlist -> Subagent kann es
  nicht aufrufen, auch wenn die Tiefe es zuliesse.
- Profile-roleDefinition sagt explizit "Do NOT switch modes or spawn
  further subagents" als Prompt-Leitplanke.

Drei Linien.

#### Settings-Default-Migration

**Status:** Robust.

`NewTaskTool` liest `this.plugin.settings.advancedApi?.subtaskTokenBudget`
mit Optional-Chaining UND einer Konstanten-Fallback
(`DEFAULT_SUBTASK_TOKEN_BUDGET = 8000`). Eine alte data.json ohne den
neuen Setting-Eintrag faellt sauber auf 8000 zurueck, kein NaN, kein
Spawn-ohne-Budget. Mirrors den Default in `settings.ts`.

#### Subtask-Token-Akkumulation in Parent-Totals

**Status:** Korrekt + nicht-security-relevant.

`spawnSubtask` callback akkumuliert subtask-Tokens in
`totalInputTokens` etc. via die `onUsage`-Forwards. Das ist fuer die
UI-Cost-Anzeige korrekt (User sieht Gesamtkosten); kein Doppel-Zaehlen
mit dem Parent-Loop, weil der `onUsage`-Callback nur einmal pro
Subtask-Anfrage feuert.

## SCA

Keine neuen Runtime-Dependencies durch FEAT-24-04. `npm audit`-Baseline
unveraendert gegenueber AUDIT-020 (1 vorbestehender Moderate `mermaid`,
DEBT-SCA-2026-05-12).

## Positivbefunde

- **Klare, mehrfach gesicherte Vertrauensgrenze** beim Profile-Spawn:
  Allowlist + mcpClient-Suppression + Profile-roleDefinition-Prompt-
  Leitplanke + maxSubtaskDepth.
- **Per-Call-Token-Budget greift vor jedem Spawn** (Tier-4 + Profile);
  Settings-Default robust mit Konstanten-Fallback fuer alte data.json.
- **Profile-Allowlist gewinnt auch gegen `find_tool`-Aktivierung**
  (struktureller Beweis durch Order-Of-Filters in rebuildPromptCache).
- **Profile-roleDefinition enthaelt explizite Negativ-Anweisungen**
  (no writes, no mode-switching, no nesting, must call attempt_completion).
- **Regression-Schutz:** der `subagent-profiles.test.ts`-Test prueft
  explizit, dass keine write/edit/delete/use_mcp_tool/new_task-Namen
  in der research-Allowlist auftauchen.
- **Profile-Definition Code-konstant** -- kein User-Eingabe-Vektor
  heute, kein Prompt-Injection-Surface. F-1 dokumentiert den
  Pre-Emptive-Hinweis fuer Folge-Items.
- **Konsistenz zum ADR-090-Schutz:** non-profile-Pfad unveraendert
  (Tier-4-Justification weiter required). Der Profile-Pfad ergaenzt,
  ersetzt nicht.
- **Subagent-Outputs unterliegen FEAT-24-02-Microcompaction** wie jedes
  andere Tool-Result; keine besondere Behandlung noetig.

## Architektonische Folgepunkte

Keiner kritisch fuer FEAT-24-04. F-1 (user-eingebbare Profile als
zukuenftiger Injection-Vektor) ist ein Pre-Emptive Note fuer kuenftige
Erweiterungen, kein heutiges Item. Mittelfristig waere es sinnvoll,
weitere Profile (z.B. `summarise`, `code-review`) zu addieren, sobald
der Live-Messlauf zeigt dass `research` real genutzt wird.
