### P3: Consider (Low + Info)

#### L-1: createNodeFetch has no auto-timeout (zero-trust)

| Field | Value |
|-------|-------|
| Severity | Low |
| CWE | CWE-400 (Uncontrolled Resource Consumption) |
| Location | `src/api/providers/openai.ts:76-143` |
| Status | **Resolved** (2026-05-13 fix-loop: 120 s socket idle-timeout added; cleared once the server starts streaming via `req.setTimeout(0)` in the response callback) |

The Node.js `http(s).request` call inside `createNodeFetch` does not set a
socket timeout. A misbehaving server (e.g. a custom OpenAI-compatible server
that accepts the connection and then never writes a response) keeps the
socket open until the user aborts via the chat-level cancel.

The path FIX-04-03-03 adds three more provider types (`custom`, `ollama`,
`lmstudio`) on top of the existing `gemini` route. The user can cancel
each stuck request via `AbortSignal` (wired through line 133-135 and the
existing chat-level Stop button), so denial of service is bounded by the
user, not by an external actor.

**Remediation:** add `req.setTimeout(120_000, () => { req.destroy(...); })`
(two-minute idle-socket timeout, same order as the chat-level rateLimitMs
maximum) so a stuck server hangs at most two minutes before the connection
drops on its own.

**Effort:** S. Single helper change in `createNodeFetch`.

#### L-2: ListPinnedConversationsTool surfaces SQLite error text (information disclosure)

| Field | Value |
|-------|-------|
| Severity | Low |
| CWE | CWE-209 (Generation of Error Message Containing Sensitive Information) |
| Location | `src/core/tools/vault/ListPinnedConversationsTool.ts:81-84` |
| Status | **Resolved** (2026-05-13 fix-loop: catch-block writes raw `e` to `console.warn`, returns generic `Failed to query pinned conversations.` to the tool result; regression test updated) |

The catch block writes the raw exception message into the tool result:

```ts
ctx.callbacks.pushToolResult(this.formatError(
    `Failed to query pinned conversations: ${e instanceof Error ? e.message : String(e)}`
));
```

A SQL error could surface column or table names, which gives the LLM a
schema hint it would not otherwise have. The user already has full read
access to the local memory database, so this is closer to "diagnostic
verbosity" than "secret leak".

**Remediation:** keep the raw message in `console.warn` for the developer
console and return a generic `Failed to query pinned conversations.` to
the tool result. Matches the pattern used in `main.ts:2519`
(`countMemoryFactsForConversation`).

**Effort:** S. Two lines.

#### L-3: createNodeFetch widens server-side request surface to user-chosen URLs (zero-trust / SSRF-shaped)

| Field | Value |
|-------|-------|
| Severity | Low |
| CWE | CWE-918 (Server-Side Request Forgery) |
| Location | `src/api/providers/openai.ts:180-192` |
| Status | Deferred to backlog as IMP-04-03-05 (Models-tab warning when a Custom provider points at a non-loopback http:// URL) |

After FIX-04-03-03, when a user selects provider type `custom`, `ollama`,
or `lmstudio`, the OpenAI SDK routes through `createNodeFetch`. The user-
controlled `baseUrl` then becomes the destination of Node.js `http(s)`
requests issued from the Electron renderer. Two ways this matters:

1. **Internal-network reachability.** The renderer can now reach hosts
   that the browser-side fetch could not reach (no CORS preflight). A user
   who types `http://192.168.0.1/admin` as a base URL hits that host on
   their LAN, with API-key headers attached. The renderer itself is the
   sole initiator: there is no remote attacker injecting a URL.
2. **TLS validation still applies for `https://`.** Node's default agent
   does verify certs. The new HTTP path (port 80, plain) only fires when
   the user explicitly types `http://`.

**Remediation:** none required for v2.7.4. If we later need to harden
this we can warn in the Models tab when a `custom` provider points at a
non-loopback HTTP URL ("Sending chat history in plain text to a remote
host. Switch to https or confirm.") and let the user override per request.

**Effort:** M when implemented (UI text plus warning gate).

#### L-4: FactExporter does not escape backslashes (code-scanning #66)

| Field | Value |
|-------|-------|
| Severity | Low |
| CWE | CWE-20 (Improper Input Validation), CWE-116 (Improper Encoding) |
| Location | `src/core/memory/FactExporter.ts:148` |
| Status | **Resolved** (2026-05-13 fix-loop: backslash-escape added BEFORE backtick-escape so the second pass cannot double-process the inserted backslash) |
| Source | GitHub code-scanning alert #66 (`js/incomplete-sanitization`) |

`escapeMarkdown` strips newlines and escapes backticks but leaves
backslashes in place:

```ts
function escapeMarkdown(text: string): string {
    return text.replace(/\r?\n/g, ' ').replace(/`/g, '\\`');
}
```

A fact value that ends with `\` followed by a backtick would emit
`\\``: the first backslash escapes the second, so the backtick stays
unescaped and breaks out of the inline-code span in the bullet.

**Risk:** Low. The exported markdown is consumed by the user themselves
(or by the LLM reading the export), not by a browser parsing untrusted
HTML. The worst case is malformed markdown rendering, not script
execution. But CWE-116 is well-known and the fix is one line.

**Remediation:** escape backslashes first, then backticks:

```ts
function escapeMarkdown(text: string): string {
    return text
        .replace(/\r?\n/g, ' ')
        .replace(/\\/g, '\\\\')
        .replace(/`/g, '\\`');
}
```

**Effort:** S.

#### I-8: ConversationStore.ts:92 Math.random() fallback (code-scanning #67, false-positive)

| Field | Value |
|-------|-------|
| Severity | Info |
| Location | `src/core/history/ConversationStore.ts:92-94` |
| Status | False positive in practice; documented |
| Source | GitHub code-scanning alert #67 (`js/insecure-randomness`) |

CodeQL flags the ternary fallback:

```ts
const uuid = (typeof crypto !== 'undefined' && crypto.randomUUID)
    ? crypto.randomUUID().replace(/-/g, '').slice(0, 12)
    : Math.random().toString(36).slice(2, 14);
```

CodeQL sees only the `Math.random()` branch and warns. In the Electron
renderer (the only runtime where this code runs), `crypto.randomUUID`
is always defined, so the fallback is dead in practice. AUDIT-016 M-4
documented this trade-off: keep a fallback to avoid hard-crashing on
exotic embeds; rely on randomUUID for the entropy story. The first
branch covers 122 bits of entropy plus 48 hex chars after slicing.

**Action:** none. Add a CodeQL suppression comment next time we touch
the file. Not worth a standalone change.

| Field | Value |
|-------|-------|

## Positive Findings

- The trust-boundary of `update_settings` was preserved when the allowlist
  was expanded (I-1).
- All new external surfaces (`list_pinned_conversations`,
  `read_mcp_tool`) reuse existing allowlists (`activeMcpServers`,
  `WRITABLE_PATHS`) instead of opening fresh ones (I-2, I-4).
- The deferred-tool guard fails closed and gives the model an
  actionable next step (`call find_tool`) rather than silently rejecting
  (I-3).
- The research subagent profile maintains the no-write contract through
  the allowlist; the prompt rewrite cannot grant new capabilities (I-5).
- The MemorySourceStore second-pass init is idempotent; even if the
  init path runs more than once, only one store instance lives in
  `plugin.memorySourceStore` (I-6).
