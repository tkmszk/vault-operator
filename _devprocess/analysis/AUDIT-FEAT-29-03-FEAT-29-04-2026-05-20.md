## Findings

### M-1 (Medium): NoticeCapture Monkey-Patch Race-Condition

**CWE-362 Concurrent Execution using Shared Resource with Improper Synchronization**

**Location:** `src/core/utils/NoticeCapture.ts:79-179`

**Risk:** `withNoticeCapture` patches `globalRef.Notice` global. Wenn waehrend des `tailMs`-Windows (default 250ms) ein zweiter Caller `withNoticeCapture` startet, liest dieser nicht das echte `OriginalNotice` sondern den bereits gepatchten Constructor. Der Fluss:

1. Caller A: liest `OriginalNotice` = echter Notice, ersetzt mit `PatchedA`, startet fn.
2. fn returnt, A geht in tailMs-Window.
3. Caller B startet waehrend tailMs: liest `OriginalNotice` = `PatchedA`, ersetzt mit `PatchedB`, startet seine fn.
4. Notices in B's fn-Window gehen durch `PatchedB` -> recordNotice in B's Array -> Delegate an `PatchedA` -> recordNotice in A's Array (das ist noch im Memory referenziert, aber A's tailMs-Promise haengt noch!)
5. A's tailMs-Window endet, A's finally restoriert "OriginalNotice" = der echte Notice global.
6. B haengt noch in seinem fn / tailMs-Window. B's `globalRef.Notice` ist jetzt der echte Notice. Aber B's finally restoriert spaeter "OriginalNotice" = `PatchedA` (das ist B's Memory-Snapshot vom Start) -> der echte Notice ist wieder weg, jetzt zeigt globalRef.Notice auf `PatchedA`, und der zeigt auf ein verwaistes A-Array das nie mehr gelesen wird.

**Folgen:**
- Notices nach Welle 3 koennen in einem verwaisten Array landen (Memory-Leak / silent data-loss)
- Die Notice-API ist nach diesem Race auf `PatchedA` festgesetzt, d.h. JEDER zukuenftige Notice-Call ueberall im Plugin geht durch ein altes recordNotice -> Memory-Leak waechst

**Threat-Model:** Im aktuellen Code-Flow ist Tool-Execution sequentiell -- ein zweiter `executeCommandById` waehrend des tailMs ist nicht erwartet. ABER: Andere Code-Pfade koennten parallel laufen (z.B. ExecuteRecipeTool das intern Commands feuert, MCP-Server-Calls, ein Hintergrund-Recipe-Executor). Plus das fenster ist 250ms, das ist lange genug fuer einen second-tool-call.

**Realismus:** Niedrig-bis-Mittel. Im sequentiellen Tool-Pipeline-Modus tritt das selten auf. Aber wenn es eintritt, ist der Schaden persistent (jeder spaetere Notice korrupt) und schwer zu debuggen.

**Remediation:** Token-basierte State-Machine. Nur einmal global patchen, Nested-Caller koennen entweder (a) ihre Notices in den selben Bus schreiben, oder (b) sofort fail-soft mit `patchSkipped=true` zuruecksetzen:

```typescript
// Module-level singleton state
let activePatch: { token: symbol; original: unknown } | null = null;

export async function withNoticeCapture<T>(
    globalRef: { Notice?: unknown },
    fn: () => Promise<T> | T,
    options: NoticeCaptureOptions = {},
): Promise<NoticeCaptureResult<T>> {
    // If a patch is already active (race), this caller runs without capture.
    if (activePatch !== null) {
        const result = await fn().catch((e) => {
            return { __error: e instanceof Error ? e : new Error(String(e)) };
        });
        return {
            result: ((result as { __error?: Error }).__error ? undefined : result) as T,
            notices: [],
            truncated: false,
            capturedError: (result as { __error?: Error }).__error ?? null,
            patchSkipped: true,
        };
    }
    // ... existing patch logic, only this caller writes to activePatch
}
```

**Status:** Resolved (2026-05-20).

### L-1 (Low): probe_plugin reflectApiMethods bricht bei Property-Getter mit Side-Effect

**CWE-754 Improper Check for Unusual or Exceptional Conditions**

**Location:** `src/core/tools/agent/ProbePluginTool.ts:146-152`

**Risk:** `reflectApiMethods` macht in der for-Schleife `apiHolder[key]`. Bei einem Plugin das eine seiner Properties als Getter implementiert ist und im Getter throws (z.B. "API not initialized"), bricht die ganze Schleife mit Crash. probe_plugin returnt dann einen Error statt einer leeren / partial api-methods-Liste.

**Threat-Model:** Reines Reflection-Verhalten. Plugin-Bugs koennten unsere Probe brechen. Kein Sicherheits-Risiko aber Functional-Defekt.

**Realismus:** Niedrig. Die meisten Plugins exposen einfache `function`-Properties ohne Getter-Magic.

**Remediation:** try/catch um den Property-Access. Bei Error den Property einfach skippen:

```typescript
for (const key of Object.keys(apiHolder)) {
    if (key.startsWith('_')) continue;
    if (PLUGIN_BASE_METHODS.has(key)) continue;
    let value: unknown;
    try {
        value = apiHolder[key];
    } catch {
        continue; // getter throw -- skip
    }
    if (typeof value !== 'function') continue;
    out.push(key);
}
```

**Status:** Resolved (2026-05-20).

### L-2 (Low): NoticeCapture per-Notice keine Laengen-Begrenzung -> tool_result-Bloat

**CWE-770 Allocation of Resources Without Limits or Throttling**

**Location:** `src/core/utils/NoticeCapture.ts:107-124`

**Risk:** `recordNotice` speichert `text` ohne Laengen-Limit. Ein Plugin das eine 50KB Stack-Trace als Notice schreibt wuerde 50KB ins tool_result eines einzelnen Notices schreiben. Bei 100 Notice-Calls (max-Capture) sind das bis zu 5MB pro Tool-Call. tool_result wandert in die Conversation-History, das frisst Context-Window.

**Threat-Model:** Akzeptable-Plugins schreiben kurze Notices. Aber Debug-Plugins, Logger-Plugins oder buggy Plugins koennten massive Notices schreiben. Folge: LLM-Context-Bloat, hoehere Token-Kosten, mediocre UX.

**Realismus:** Niedrig. Die meisten Notices sind < 200 chars. Aber Sebastian-spezifisch (Dataview-Query-Results, Templater-Output) koennten gross werden.

**Remediation:** Per-Notice-Text-Truncation auf z.B. 500 chars, mit Marker:

```typescript
const MAX_NOTICE_TEXT_CHARS = 500;
const rawText = ...;
const text = rawText.length > MAX_NOTICE_TEXT_CHARS
    ? rawText.slice(0, MAX_NOTICE_TEXT_CHARS) + '... [truncated]'
    : rawText;
```

**Status:** Resolved (2026-05-20).

### I-1 (Info): NoticeCapture Sensitive-Heuristik deckt nur explicit "key/token/secret/password"-Woerter ab

**Location:** `src/core/utils/NoticeCapture.ts:59`

**Risk:** Pattern `\b(token|secret|key|password|api[-_ ]?key)\b/i` matched nur explizite Wort-Vorkommen. False-negatives (sensible Daten die NICHT redacted werden):

- "Bearer eyJ..." -- "Bearer" matched nicht
- "PAT ghp_xyz..." -- "PAT" und "ghp_" matched nicht
- "ssh-rsa AAA..." -- matched nicht
- "Authorization: ..." -- matched nicht
- "abc123" als naked Token-String -- matched nicht

False-positives (harmless Texte die redacted werden):
- "Pressed key Escape" -- "key" matched
- "Public key cryptography" -- "key" matched
- "Master password manager open" -- "password" matched

**Threat-Model:** Sensitive-Heuristik ist first-line-defense. Bei legitimen Plugin-Notices ist die Wahrscheinlichkeit, dass ein Token-String tatsaechlich im Notice steht, gering. Plugins SOLLTEN keine Secrets in UI-Notices schreiben. Trotzdem ist das eine Defense-in-Depth-Schicht.

**Realismus:** Niedrig. Plugin-Notices enthalten typischerweise keine Tokens. Defense-in-Depth.

**Remediation (optional):** Erweitere Pattern um Token-Format-Detection:

```typescript
const SENSITIVE_PATTERN_KEYWORDS = /\b(token|secret|key|password|api[-_ ]?key|bearer|pat|auth)\b/i;
const SENSITIVE_PATTERN_TOKEN_FORMATS = /\b(ghp_[A-Za-z0-9]{20,}|sk-[A-Za-z0-9_-]{20,}|eyJ[A-Za-z0-9._-]{20,})\b/;
function isSensitive(text: string): boolean {
    return SENSITIVE_PATTERN_KEYWORDS.test(text) || SENSITIVE_PATTERN_TOKEN_FORMATS.test(text);
}
```

**Status:** Resolved (2026-05-20). Pattern erweitert um Token-Formate (ghp_, gho_, github_pat_, sk-, eyJ, generische 32+ hex) und keyword bearer/pat/auth(orization). Test "detects naked token formats without keyword" verifies das.

## Positive Findings

1. **withNoticeCapture restoriert in finally**: Selbst bei Exception in fn ist der Notice restored.
2. **try/catch um delegierten OriginalNotice-Call** schuetzt vor Plugin-Constructor-Errors (Notice-Konstruktor-Throw bricht nicht das Tool).
3. **try/catch um recordNotice** schuetzt davor dass eine boese Notice-Argument-Type den Capture bricht.
4. **PatchedNotice prototype-mirroring** preserved `instanceof Notice` checks in Plugins.
5. **maxCaptures-Truncation** verhindert unbounded list-growth.
6. **patchSkipped Fail-Soft** wenn Notice-Konstruktor nicht patchable (Test-Env, nicht-DOM).
7. **probe_plugin ist read-only** (isWriteOperation=false), keine Approval-Pflicht.
8. **probe_plugin Whitelist-Filter** (PLUGIN_BASE_METHODS) verhindert dass Obsidian-Plugin-Base-Methoden als API-Methoden gemeldet werden.
9. **probe_plugin Private-Filter** (`_`-Prefix) respektiert die Convention.
10. **ExecuteCommandTool gibt JSON-strukturiert zurueck** -- Agent kann programmatisch reagieren statt aus Freitext zu parsen.
11. **Severity-Heuristik via Regex** ist transparent und reviewable.
12. **VaultDNAScanner-Polling auf 2s + 200ms-Debounce-Event-Hook** in main.ts ist UI-responsive ohne CPU-Druck.

## Fix-Loop -- alle Findings resolved

**M-1 (Race-Protection):** Module-level `activePatch`-Singleton mit Symbol-Token. Erster Caller patcht; Nested-Caller waehrend des tailMs-Windows laufen fail-soft (kein Re-Patch, `patchSkipped=true`, leeres notices-Array). Token-Check beim Cleanup verhindert dass ein Nested-Caller den Patch des Outer-Callers vorzeitig restored. 2 neue Tests: "runs a nested withNoticeCapture fail-soft" + "clears the singleton activePatch after fn settles".

**L-1 (Getter-Tolerance):** try/catch um den `apiHolder[key]`-Access in `reflectApiMethods`. Bei Getter-Throw wird das Property skipped statt zu crashen. 1 neuer Test "skips properties whose getter throws".

**L-2 (Per-Notice-Truncation):** Neue Konstante `MAX_NOTICE_TEXT_CHARS = 500`. `rawText.slice(0, 500) + '... [truncated]'` bei zu langem Text. 1 neuer Test "truncates per-notice text".

**I-1 (Sensitive-Pattern erweitert):** Pattern aufgeteilt in zwei Regexes:
- `SENSITIVE_PATTERN_KEYWORDS` -- ergaenzt um `bearer`, `pat`, `auth(orization)?`
- `SENSITIVE_PATTERN_TOKEN_FORMATS` -- erfasst `ghp_*`, `gho_*`, `github_pat_*`, `sk-*`, `eyJ*` (JWT), generische 32+ hex
- `isSensitiveText()` als zentraler Detector. 1 neuer Test "detects naked token formats without keyword".
