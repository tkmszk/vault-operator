## Findings

### L-1: Stale `manage_skill` references after tool removal -- RESOLVED

**Severity:** Low (Dead-Code-Cleanup, no exploitable surface)
**CWE:** CWE-561 Dead Code (defense-in-depth)
**Location:** Multiple files
**Status:** Resolved im Audit-Pass

**Risk:** Nach Removal des manage_skill-Tools verblieben stale String-Referenzen in:
- `src/core/tool-execution/ToolExecutionPipeline.ts:146` -- TOOL_GROUP_MAP-Eintrag (group: self-modify)
- `src/core/tool-execution/ToolExecutionPipeline.ts:566` -- Kommentar
- `src/core/tool-execution/ResultExternalizer.ts:40` -- in TOOLS_TO_NOT_EXTERNALIZE-Liste
- `src/core/mastery/SuggestionService.ts:105` -- actionHint-String "manage_skill create with steps based on: ..."
- `src/mcp/McpBridge.ts:32` -- AGENT_INTERNAL_TOOLS-Set
- `src/mcp/tools/__tests__/executeVaultOp.test.ts:90` -- it.each-Array

Praktisches Risiko: niedrig. Die Strings sind data-only, kein Tool-Call. Aber: wenn der Agent oder UI die Strings irgendwo durchreicht und der Pipeline-Lookup `TOOL_GROUP_MAP['manage_skill']` zurueckkommt → ein nicht-existentes Tool wuerde fuer "self-modify-Approval" gebucht. Dead path, aber sauberes Aufraeumen schliesst die Falle.

**Remediation (applied):**
- ToolExecutionPipeline.ts: `manage_skill: 'self-modify'` Zeile entfernt, Kommentar angepasst
- ResultExternalizer.ts: `'manage_skill'` aus Array entfernt
- SuggestionService.ts: actionHint zeigt auf skill-creator-builtin Workflow
- McpBridge.ts: `'manage_skill'` aus AGENT_INTERNAL_TOOLS entfernt
- executeVaultOp.test.ts: `'manage_skill'` aus it.each-Array entfernt mit erklaerendem Kommentar

### L-2: SandboxBridge.vaultMkdir defense-in-depth -- BY DESIGN

**Severity:** Low (already defense-in-depth)
**Location:** `src/core/sandbox/SandboxBridge.ts:vaultMkdir`
**Status:** by design

**Risk:** Neue Methode `vaultMkdir(path)` walked Pfad-Segmente und mkdir'd. Path-Traversal? Configdir-Bypass?

**Analysis:**
- `validateVaultPath(p, isWrite=true)` blockt `..`, `/`, `\` und configDir-prefix (= .obsidian)
- Test 1: `vaultMkdir('../escape')` -> wirft "Invalid path" ✓
- Test 2: `vaultMkdir('.obsidian/plugins/evil')` -> wirft "configDir/ is protected" ✓
- Test 3: Idempotent wenn Folder existiert ✓

Pin: Tests in `src/core/sandbox/__tests__/SandboxBridge.hiddenPaths.test.ts > vaultMkdir`. No action needed.

### L-3: init_skill.js relPath construction -- BY DESIGN

**Severity:** Low (already defense-in-depth, two-layered)
**Location:** `bundled-skills/skill-creator/scripts/init_skill.js`
**Status:** by design

**Risk:** Das Sandbox-Skript schreibt `${skillRoot}/SKILL.md`, `${skillRoot}/scripts/.gitkeep` etc. Wenn `name` boes ist (`..`, absolute path, NUL byte), kann das in einen anderen Pfad schreiben?

**Analysis (two-layer defense):**
1. Im Skript: `validateName(name)` mit Whitelist-Regex `[a-z0-9][a-z0-9-]*[a-z0-9]` oder `[a-z0-9]`. Reserved words + max 64 + double-hyphen-check. Plus Layer:
2. In SandboxBridge.vaultWrite: `validateVaultPath(path, isWrite=true)` blockt `..`, `/`, `\`, configDir-Prefix.

Beide Layers muessten umgangen werden. Plus FEAT-29-11-Audit-L-1-Fix in `BuiltinSkillMaterializer` zeigt das Pattern: relPath wird im production-code immer durch isSafePathSegment validated. Konservativ, kein direkter Patch noetig.

### I-1: RerankerService globalThis Symbol-Prepop -- BY DESIGN

**Severity:** Info
**Location:** `src/core/knowledge/RerankerService.ts:67-78`
**Status:** by design

**Risk:** Wir setzen `globalThis[Symbol.for("onnxruntime")] = onnxruntime-web-module` vor dem transformers-import. Falls ein anderes Plugin/Modul bereits einen Wert in dieses Symbol-Slot gelegt hat (z.B. eine fake-runtime), wuerde transformers.js den nehmen und Inferenz darin ausfuehren.

**Justification:** Symbol.for-Slots in globalThis sind theoretisch ueber Plugin-Grenzen hinweg sichtbar in Obsidian. Aber: das Plugin-Ecosystem ist explizit User-trusted (User installiert was er will). Wenn ein anderes Plugin diesen Slot vor uns belegt, hat der User dem Plugin die Berechtigung gegeben das zu tun. Threat-Model ausserhalb des Plugin-Scope.

Wir koennten den `if (!(ortSymbol in globalThis))` durch `globalThis[ortSymbol] = ort` ohne Check ersetzen (force-override). Aber das ist agressiver, koennte legitime andere Anwendungsfaelle brechen (z.B. ein anderes Plugin das auch transformers nutzt). Aktueller Code ist defensiv-richtig.

### I-2: VaultDNAScanner.toKebabCase output for pathological input -- INFO

**Severity:** Info
**Location:** `src/core/skills/VaultDNAScanner.ts:toKebabCase`
**Status:** Tolerable

**Risk:** Input `"___"` -> Output `""` (alle Zeichen rausgestrippt). Resultierende SKILL.md mit `name: ` (leer) wuerde vom Validator rejected, aber das ist ein Validator-Job, nicht ein Konvertierungs-Bug.

**Justification:** Plugin-IDs die nur aus Sonderzeichen bestehen sind im Obsidian-Plugin-Ecosystem nicht moeglich (Obsidian's Plugin-Manifest-Schema validiert das selbst). Pathological input erscheint praktisch nie. Validator-Reject hat als Fallback aus.

### I-3: ModelDiscoveryService + SemanticIndex log-level changes -- INFO

**Severity:** Info
**Location:** `src/core/routing/ModelDiscoveryService.ts`, `src/core/semantic/SemanticIndexService.ts`
**Status:** Observability

**Note:** Network-level errors (Ollama daemon ECONNREFUSED) und corrupted-zip-archives wurden von `console.warn` auf `console.debug` heruntergestuft. Das ist Observability-Polish, kein Security-Issue. User hat UI-Feedback (Settings-leerer-Models-Liste, fehlende-Index-Hits) als Indikator.

## Positive Findings

- **SkillFrontmatterValidator** -- Whitelist-Regex `[a-z0-9][a-z0-9-]*[a-z0-9]` ist ReDoS-safe (kein nested-quantifier, lineare Komplexitaet). Reserved-words `anthropic`/`claude` explizit gegen Spoofing.
- **SandboxBridge.isHiddenPath** -- Dot-Segment-Detection findet hidden folders auch in nested paths (`Notes/.archive/...`). Greift symmetrisch fuer read/write/list/mkdir.
- **EditFileTool adapter-fallback** -- gleiche Pattern wie WriteFileTool/ReadFileTool, kein neuer Spruch in der Architektur.
- **init_skill refuses to overwrite** -- explizite Error-Message mit recovery-Sequence ("call delete_folder(...) first") verhindert silent data loss.
- **manage_skill removal vollstaendig** -- nach Cleanup keine restlichen Code-Pfade, die das nicht-existente Tool referenzieren.
