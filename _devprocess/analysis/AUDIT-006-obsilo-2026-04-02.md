## Findings (nach Prioritaet)

### P1: Must Fix (Critical + High)

#### H-1: MCP Local HTTP Server -- Keine Authentifizierung + CORS Wildcard

| Field | Value |
|-------|-------|
| **Severity** | High |
| **CWE** | CWE-306 (Missing Authentication), CWE-942 (Permissive CORS) |
| **OWASP** | A01 Broken Access Control, A07 Authentication Failures |
| **Location** | `src/mcp/McpBridge.ts:260-314` |
| **Status** | **Resolved** (2026-04-02) |

**Risk:** Der lokale MCP-Server auf `127.0.0.1:27182` akzeptierte alle POST-Requests
ohne jede Authentifizierung. Gleichzeitig war `Access-Control-Allow-Origin: *` gesetzt.

**Fix:**
- Bearer-Token-Auth: Auto-generierter Token bei erstem Start, gespeichert verschluesselt
  in Settings, Token-File unter `~/.obsidian-agent/mcp-token` fuer mcp-server-worker
- CORS eingeschraenkt auf `app://obsidian.md` (blockiert Browser-Cross-Origin)
- Body-Size-Limit: 1 MB (M-4 mitgefixt)
- mcp-server-worker liest Token und sendet `Authorization: Bearer` Header

#### H-3: Permission Self-Escalation via update_settings

| Field | Value |
|-------|-------|
| **Severity** | High |
| **CWE** | CWE-269 (Improper Privilege Management) |
| **OWASP LLM** | LLM08 Excessive Agency |
| **Location** | `src/core/tools/agent/UpdateSettingsTool.ts:17-73`, `src/core/tool-execution/ToolExecutionPipeline.ts:84,356` |
| **Status** | **Resolved** (2026-04-02) |

**Risk:** `update_settings` war als Gruppe `'agent'` klassifiziert (immer auto-approved)
und konnte autoApproval-Flags aendern.

**Fix:** Sonder-Check in `checkApproval()`: Wenn `update_settings` mit
`action: 'apply_preset'` oder `path.startsWith('autoApproval.')` aufgerufen wird,
wird User-Approval angefordert (Approval-Card). Fail-Closed wenn kein Callback.
Alle anderen agent-Tool-Calls bleiben auto-approved.

#### M-1: Path-Traversal fehlt in Vault-Tools

| Field | Value |
|-------|-------|
| **Severity** | Medium |
| **CWE** | CWE-22 (Path Traversal) |
| **Location** | `src/core/tools/vault/WriteFileTool.ts:51`, `ReadFileTool.ts:73`, `AppendToFileTool.ts:58`, `MoveFileTool.ts:36`, `CreateFolderTool.ts:32` |

**Remediation:** Shared `validateVaultPath()` Utility extrahieren (Pattern aus
`SandboxBridge.ts:192-206`) und in allen Vault-Tools vor der Operation aufrufen.

## Positive Findings

Die Codebase zeigt fortgeschrittene Security-Awareness:

1. **Multi-Layer Sandbox:** AstValidator + iframe/Process-Isolation + SandboxBridge
   (Path-Validation, Write-Size-Limits, Rate-Limiting, URL-Allowlist, Circuit-Breaker,
   Prototype-Pollution-Check, configDir-Write-Block). Branchenfuehrende Sandbox-Architektur.

2. **Fail-Closed Defaults:** Pipeline verweigert bei fehlendem Approval-Callback.
   IgnoreService verweigert bei nicht-geladenem State. Alle Security-Features opt-in.

3. **Self-Modification Protection:** `manage_source` und `manage_skill` erfordern
   immer Manual Approval -- kein Auto-Approve-Bypass moeglich.

4. **SafeStorageService:** API-Keys per OS-Keychain verschluesselt (macOS Keychain,
   Windows DPAPI, Linux libsecret).

5. **SSRF-Schutz:** Zwei-Phasen-Check (Hostname + DNS-Resolution) mit IP-Range-Blocking.
   TOCTOU dokumentiert und acknowledged.

6. **Audit Trail:** OperationLogger mit Parameter-Sanitization (Sensitive Keys redacted,
   Content truncated, URL-Credentials stripped). 30-Tage-Rotation.

7. **Checkpoint System:** Pre-Write-Snapshots fuer granulares Rollback.

8. **Recipe Security:** 7-Layer Defense-in-Depth (Master Toggle, Per-Recipe Toggle,
   Parameter-Validation, shell:false, CWD-Confinement, Timeout+SIGKILL, Output-Limits).

9. **SQL Injection:** Alle sql.js Queries nutzen konsistent parametrisierte Statements.

10. **XSS:** Kein innerHTML in der gesamten Codebase. Durchgaengig Obsidian DOM API.

11. **Review-Bot Compliance:** console.log/info eliminiert, requestUrl statt fetch,
    ES imports, vault.configDir statt hardcoded .obsidian.

12. **Relay Security:** Constant-Time Token-Vergleich, HTTPS-Enforcement, Queue-Limits,
    Body-Size-Limits, Correlation-ID-Randomisierung.
